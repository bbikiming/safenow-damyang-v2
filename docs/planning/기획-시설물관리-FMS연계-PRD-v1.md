# PRD — 시설물 안전관리 (FMS 시설물관리대장 연계) 대메뉴 신설 v1

- 작성일: 2026-07-13
- 배경: 담양군 재난안전과가 국토안전관리원 FMS(시설물통합정보관리시스템) 표준연계서비스로부터 담양군 소관 시설물관리대장을 파일(`88384_2.xls`)로 수령. 이 원천 데이터를 중대재해 통합관리 시스템으로 불러와 저장·업데이트하고, 후속으로 위험성평가·점검·개선조치까지 연결하기 위한 대메뉴를 신설한다.
- 원칙: 기획 순수성 원칙 준수 — 구현 코드·DDL·프레임워크는 기술하지 않고, 기획 요소·데이터 항목·유효성·화면 플로우·수용 기준만 규정한다. 단 화면 ID·파일명·NAV 삽입 위치는 프로토타입 산출물 수준으로 지정한다.
- 근거 자료(검증 완료): FMS 표준연계서비스 개발자 가이드 v4.1(2019.6.3), 실데이터 80건 프로파일링, 중대재해처벌법·시설물안전법 조문(법제처 국가법령정보센터 1차 출처), 프로토타입 코드베이스 역공학.

> 중요 전제: 이 문서는 하드코딩 시뮬레이션 프로토타입 기준이다. 실제 FMS 연계(수신·전송)는 별도 이용신청·인증·표준연계모듈 적용이 선행되어야 하며, 본 대메뉴는 그 연계가 완료됐을 때의 화면·데이터·플로우를 정의한다. 프로토타입에서는 파일 업로드(엑셀) 경로를 1차 구현으로 한다.

---

## 0. 요약

- **무엇을**: 안전보건관리체계와 병렬하는 신규 대메뉴 **`시설물 안전관리`(GNB id `facil`)** 를 만들어, FMS 시설물관리대장 80건을 `FACIL_NO`(시설물번호)를 기본키로 불러와 저장하고, FMS에 없는 위험성평가용 정보를 보완입력으로 채우며, FMS로 되돌려 저장·수정하는 연계를 운영한다.
- **왜 필요한가**: (1) 현재 프로토타입은 시설물 데이터를 전부 가상 하드코딩으로 갖고 있고([base-targets.html](base-targets.html)·[js/bgt.js](js/bgt.js)·[js/proc-cascade.js](js/proc-cascade.js)), FMS 실데이터 80건과 이름조차 거의 일치하지 않는다. (2) 위험성평가·예산·개선조치 사슬에 시설물 식별자를 관통시킬 안정 키(`FACIL_NO`)를 넣을 자리가 없다. (3) 받은 데이터만으로는 위험성평가를 한 건도 굴릴 수 없다 — 안전등급·점검이력·규모가 통째로 빠져 있어, 부족분을 채우는 구조가 필요하다.
- **핵심 제약**: 받은 파일은 FMS 8개 연계업무 중 **시설물관리대장 기본현황(BASTB_MASTER) 1종뿐**이다. 위험도 산정의 핵심 변수(안전등급·점검일·중대결함·규모)는 FMS의 다른 테이블 소관이며 이번 파일에 0건이다. 따라서 본 대메뉴는 "FMS 불러오기"가 아니라 "FMS 불러오기 + 위험성평가용 보완입력 + FMS 되돌려쓰기"의 3층 구조로 설계한다.

## 1. 현황 진단

### 1-1. 받은 데이터의 정체

- 파일 `88384_2.xls` 는 확장자만 xls이고 실제 형식은 xlsx다. 단일 시트 `Data`, 헤더 1행 + **80개 시설물 행**.
- 각 행은 `Ð`(U+00D0) 구분자로 나뉜 **43개 토큰**이며, 첫 토큰이 `BASTB_MASTER`(FMS 식별자/참조테이블명)이고 나머지 42토큰이 FMS 「표 1. 연계데이터 구조 – 시설물관리대장 기본현황」의 42개 컬럼과 순서까지 정확히 1:1 일치한다.
- 즉 이 파일은 임의 엑셀이 아니라 **FMS 표준연계 전송 레코드 포맷 그대로**다. 연계 질의서가 명시한 대체 경로("연계가 어렵거나 지연될 경우 FMS 시설물 목록 엑셀 제공 및 일괄 업로드")에 해당한다.

### 1-4. 개발자 가이드가 낡았다 (설계 반영 필수)

가이드는 v4.1 / 2019.6.3 판인데 실데이터와 어긋난다. **연계 규격을 이 가이드만 믿고 확정하면 안 된다.**

| 항목 | 가이드 v4.1 | 실데이터 | 판정 |
|---|---|---|---|
| 레코드 필드 수 | 40개(예시) | 42개 | 공통코드 개편 후 컬럼 추가 |
| 시설물구분/종류 | GBN=`RO`, KIND=`BR` | GBN=`BR`, KIND=`ROB` | 코드 체계 개편 |
| 시설물종별(FACIL_CLASS) | `1,2,9` | `2,3` | 값 정의 재확인 필요 |
| 설계도서보존여부(DSN_KEEP_STATUS) | `Y,N` | `1/2/3/4` | 값 정의 재확인 필요 |

→ 후속 조치(§9)로 국토안전관리원 FMS 최신 공통코드 재확인을 명시한다.

### 1-5. 프로토타입 코드베이스와의 정합

- [js/bgt.js:15](js/bgt.js:15)는 이미 `시설 원본 = 대상 관리(FMS 시설물 현황) 연계`를 전제하고 자체 FMS 동기화를 폐지했다. [js/bgt-settings.js:313](js/bgt-settings.js:313)는 "시설물 현황은 대상 관리 메뉴에서 FMS 연계로 자동 현행화"를 약속한다. **본 대메뉴가 이 약속을 실제로 이행하는 단일 출처가 된다.**
- 위험성평가 엔진은 시설물 물리 속성을 전혀 소비하지 않는다([js/proc-cascade.js:172](js/proc-cascade.js:172)의 `FACILITY`는 `dept`·`proc` 라우팅 키만 가짐). FMS 42컬럼 중 위험성평가가 현재 쓸 수 있는 것은 사실상 `FACIL_NM` 하나뿐이다.
- 프로토타입 시설물 명단은 FMS 실데이터와 불일치: [js/bgt.js](js/bgt.js) `FAC_SOURCE` 9건 중 FMS와 일치 1건, [base-targets.html](base-targets.html) 공중이용시설 표시 4건 중 일치 0건.

## 3. 화면 정의

### FAC01-V 시설물 대장 (목록)

- **목적**: FMS 수신 80건을 목록화하고 상태를 태그로 노출해 상세·위험도로 진입시킨다.
- **영역**: 상단 KPI(총 80 · 2종 29 · 3종 51 · 노후 25 · 좌표결측 80 · 소관미상 N) → 필터 바 → 목록 테이블 → 페이지네이션.
- **필터**: 시설물구분(교량/건축물/하천/상하수도/기타) · 종별(2종/3종) · 소관구분(담양 소유/국가/민간/미상) · 동기화상태 · 노후(30년 초과) · 결측(좌표/사진/소관).
- **테이블 컬럼**: 시설명(FACIL_NM) · 시설물번호(FACIL_NO) · 구분/종류 · 종별 · 소재지(ADDR 조합) · 준공(경과연수) · 소관부서(보완입력) · 동기화상태 · 관리.
- **플로우**: 행 클릭 → FAC02-D 상세. [FMS 연계] 버튼 → FAC04-S.
- **유효성/표시 규칙**: 좌표 결측 시 지도 아이콘 비활성. 소관 미상(소유자 공란 21건 + 국가/민간 10건) 행은 '소관확인 필요' 태그. FACIL_NM 중복('삼지교' 2건)은 FACIL_NO 병기로 구분.

### FAC02-D 시설물 상세 · 보완입력

- **목적**: FMS 원본 42필드를 읽기전용으로 보여주고, 위험성평가에 필요하나 FMS에 없는 항목을 자체 입력으로 채운다.
- **영역**(탭 또는 세로 구획, 단일 화면):
  1. **FMS 기본현황** — 42컬럼을 구획별(식별/위치/이력/설계·시공/코드)로 표시, **전부 읽기전용**. 결측 필드는 '—(FMS 미제공)'.
  2. **보완입력(자체 소유)** — 안전등급 · 최근 점검일 · 차기 점검예정일 · 중대결함 유무·유형 · 보수보강 진행상태 · 규모(연면적/연장/경간/처리용량) · 이용인원(일평균) · 위험물 취급 여부 · 인접 위험요소 · 좌표(지오코딩) · 소관부서(조직도 연계) · 관리담당자 · 중대재해대상 구분.
  3. **연계 이력** — 이 시설물의 수신/전송 내역(방향·인터페이스ID·반영키·필드 diff·결과·일시).
- **플로우**: [위험성평가 착수] → `risk-form.html?target=<FACIL_NO>&name=<FACIL_NM>` 로 이동해 대상 자동 선택(§5-4). [보완입력 저장] → 자체 필드만 저장(FMS 원본 불가침).
- **유효성**: 안전등급은 A~E 중 택1(시설물안전법 시행령 별표8 정의를 툴팁으로). 차기 점검예정일은 안전등급+종별로 법정주기 산출값을 기본 제시하되 수정 가능. 중대결함 '유'이면 조치 착수/완료 기한(통보일+2년/+3년) 자동 계산·표기.

### FAC03-V 시설물 위험도

- **목적**: 80건을 잠정 위험도로 스코어링·랭킹해 위험성평가 착수 우선순위 큐를 제공한다.
- **위험도 산정 변수**: FMS 확보(경과연수·종별·내진적용여부) + 보완입력(안전등급·점검경과·중대결함·이용인원). 각 시설물의 변수 확보율(예: 4/8)을 함께 표시해 '데이터 부족으로 평가 불가'를 명시.
- **영역**: 위험도 상위 랭킹 카드/테이블 + 색상 구간(높음/보통/낮음/산정불가) + [위험성평가 착수] 버튼.
- **유효성**: 안전등급·점검일 등 핵심 변수가 미입력이면 위험도 '산정불가'로 분류하고 착수 버튼 대신 [보완입력] 유도. **없는 데이터를 0으로 처리하지 않는다**(과소평가 방지).

### FAC04-S FMS 연계 (불러오기·차이검토·반영 / 전송)

- **목적**: FMS 시설물관리대장을 불러오고(수신), 자체 변경분을 FMS로 저장/수정/삭제(전송)하되, **반영 전 차이를 사람이 검토·승인**한다. 원본을 보존하고 맹목 덮어쓰기를 금지한다.
- **영역**(단일 모달 규칙 준수 — 적층 모달 금지, 같은 화면 내 스텝/인라인 패널):
  - **수신(IN)**: [파일 업로드(엑셀)] 또는 [FMS 수신] → 원본 스테이징 적재 → **차이 미리보기**(신규/변경/삭제 건, 필드별 diff) → 항목 선택 → [반영]. 프로토타입 1차는 엑셀 업로드 경로.
  - **전송(OUT)**: 로컬 변경건 목록 → 인터페이스ID별 전송(insert/update/deleteBastbMaster) → 응답 메시지코드·FMS반영키 저장.
- **플로우 유효성**:
  - 차이 미리보기에서 **자체 소유 필드(보완입력)는 FMS 수신으로 덮어쓰지 않는다**(§4 필드 오너십).
  - 전송 시 `insertBastbMaster`는 상세제원(BASTB_DTL_*) 최소 1건을 FMS가 필수 요구하므로(§5), 상세제원 미입력 상태에서는 신규 등록 전송을 차단하고 안내.
  - 수신 응답 첫 토큰이 `0000`이 아니면(E001 인증실패/E101 형식오류/E201 유효성오류/E301 서버장애) 오류 표시·재처리 큐로.
- **관계**: [admin-integration.html](admin-integration.html)(SIR-001, 전역 연계 상태 모니터)은 링크 헬스만 보고, 시설물 연계의 실제 작업·감사는 FAC04-S가 담당한다(§6).

### FAC05-S 연계 설정

- **목적**: FMS 연계 파라미터를 관리한다.
- **항목**: FMS 인증(기관코드·사용자ID·인증키·자동승인요청 기본값) · 수신 주기(일 1회 배치, 1일 실행상한 10회) · **필드 소유권 규칙**(FMS 소유/자체 소유 목록) · **공통코드 매핑표**(가이드 stale이므로 'FMS 최신 공통코드 재확인 필요' 배지) · sentinel 정규화 규칙(`x`·`확인불가` → 공란 처리).
- **유효성**: 인증정보 입력은 보안 프로토콜상 자격증명 입력에 해당 → 프로토타입에서는 표시·안내만, 실제 입력·저장은 별도 보안 절차로 위임.

## 4. 데이터 항목 정의 — 필드 소유권

FMS 되돌려쓰기(update/delete)가 FMS 정본을 오염시키지 않도록, 필드를 **FMS 소유**와 **자체 소유**로 엄격히 가른다.

### 4-1. FMS 소유 (읽기전용 · 수신 시 갱신 · 정정은 전송으로만)

- BASTB_MASTER 42컬럼 전부(FACIL_NO · FACIL_NM · MNG_MAIN_CD · 주소 4필드 · 준공일 · 설계/시공/감리 이력 · 코드값 등) + FMS반영키.
- 규칙: 화면에서 편집 불가. 수정이 필요하면 FAC04-S의 전송(updateBastbMaster) 경로로만 FMS에 반영하고, 수신으로 재확인한다.

### 4-2. 자체 소유 (편집 가능 · FMS 수신이 덮어쓰지 않음)

| 필드 | 용도 | 출처 |
|---|---|---|
| 안전등급(A~E) | 위험도 1차 결정변수 | 자체입력 or FMS 점검진단실적 연계(향후) |
| 최근 점검일 · 차기 점검예정일 | 법정주기 준수 추적 | 자체입력 or FMS 연계(향후) |
| 중대결함 유무·유형·조치기한 | 즉시위험·법정기한 | 자체입력 or FMS 연계(향후) |
| 보수보강 이력·진행상태 | 잔존위험 | 자체입력 or FMS 연계(향후) |
| 규모(연면적/연장/경간/처리용량) | 종별·공중이용시설 판정 | 자체입력 or FMS 상세제원 연계(향후) |
| 이용인원(일평균)·통과교통량 | 노출도 | 자체입력/타 시스템 |
| 위험물 취급 여부·인접 위험요소 | 위험요인 | 자체입력/현장조사 |
| 좌표(위경도) | 지도·현장동선 | 지오코딩/현장 |
| 소관부서·관리담당자 | 부서 매칭·책임 | 조직도(DYV2.ORG) 연계 |
| 중대재해대상 구분 | 스코핑 | 규모 확보 후 판정 |

### 4-3. 오너십 규칙

- 수신(IN) 반영은 4-1 필드만 갱신한다. 4-2 필드는 절대 건드리지 않는다.
- `FACIL_NO`는 전 도메인(대장→위험성평가→개선조치→이행관리→예산)을 관통하는 **안정 기본키**로 승격한다. 현재 앱은 대상을 사람이 읽는 문자열로만 보관해 안정 키가 없다(§5-4).

## 5. 동기화 모델

### 5-1. 수신(inbound)

- **프로토타입 1차**: 엑셀 업로드(`Ð` 43토큰 = 식별자 + 42컬럼 파싱) → 스테이징 → 차이검토 → 반영.
- **실연계(향후)**: FMS 수신모듈(`fmsLinkUtil.receivePostMessage`)로 시설물관리대장·유지관리실적·FMS 승인결과 일괄 다운로드. 응답 `0000|경로`이면 해당 경로에서 본문 수신. 일 1회 배치, **1일 최대 10회** 제한. 수신 자료구조(reqMethod `xxxxx@xxxx@xxx`)는 연계신청 후 국토안전관리원과 협의로 확정 — 가이드로 확정 불가.

### 5-2. 전송(outbound) — 인터페이스 ID 매핑

| 우리 시스템 행위 | FMS 인터페이스 ID |
|---|---|
| 시설물 신규 등록 | `insertBastbMaster` (+ 상세제원 BASTB_DTL_* **최소 1건 필수**) |
| 시설물 수정 | `updateBastbMaster` |
| 시설물 삭제 | `deleteBastbMaster` |
| (향후) 점검진단계획·실적 | `insert/update/deleteMantbDignPlan` · `...DignResult` |
| (향후) 보수보강계획·실적 | `insert/update/deleteMantbAmendPlan` · `...AmendResult` |
| (향후) 취약시설물 중점관리 | `insert/update/deleteMantbRestrict` |
| (향후) 중대결함 사후관리 | `updateMantbSeriousDefect` (등록/삭제 없음 — FMS가 원천, 우리는 조치만 회신) |

전송 패키지: 기관코드·사용자ID·비밀번호·인증키·자동승인요청·인터페이스ID·레코드배열·첨부(전경/정측면 사진). 응답 = 메시지코드(4자리) `|` FMS반영키 `|` 결과메시지.

### 5-3. upsert 키·충돌 정책·감사

- **upsert 키**: 시설물관리대장 = `FACIL_NO`(표57 FMS 반영키 = 'FMS 시설물번호', 채움 100%). 하위 실적은 `FACIL_NO` 컨텍스트 + 각 순번(DIGN_SEQ/AMEND_SEQ/RESTRICT_SEQ — insert 시 FMS가 채번해 회신, update/delete 시 로컬 보관 순번 재전송). 중대결함 = DEFECT_CD. → **FMS반영키 보관 슬롯 신설 필요**(현재 앱에 없음).
- **충돌 정책**: 동일 필드가 FMS·자체에서 모두 변경된 경우 — FMS 소유 필드는 FMS 우선(자체 정정은 전송으로만), 자체 소유 필드는 수신 불가침. 차이검토 화면에서 사람이 최종 승인.
- **감사추적**: 모든 수신/전송·필드 변경을 방향·인터페이스ID·시설물·반영키·필드 diff·결과·일시로 기록(FAC02-D 연계 이력 탭 + FAC04-S).

### 5-4. 위험성평가 사슬 연결 (핵심)

- `FACIL_NO`를 위험성평가 대상 키로 승격: FAC02-D/FAC03-V의 [위험성평가 착수] → `risk-form.html?target=<FACIL_NO>&name=<FACIL_NM>`. 현재 [risk-form.html](risk-form.html) 대상 option은 `data-dept/type/cdpa/grade/last`만 갖고 안정 시설 키가 없으므로, option에 `data-facil-no`(= `FACIL_NO`) 슬롯을 추가한다.
- 소관부서(자체 소유) → SIF·유해위험요인의 `applicable_dept` 매칭에 사용해, FMS에 없는 소관부서 공백을 조직도(DYV2.ORG)로 메운다.

## 7. 부족한 시설물 정보 (사용자 질문 ①)

| **안전등급(A~E)** | BASTB_MASTER에 없음. FMS 점검진단실적 STATE_GRADE 소관 | 가이드 표43 STATE_GRADE / 파일 내 점검진단실적 0건 | FMS 점검진단실적 연계 추가 수신 or 자체입력 |

| **규모**(연면적·연장·경간·처리용량·층수) | 없음. 상세제원 BASTB_DTL_* 소관 | 가이드 표2~40 / 파일 내 상세제원 0건 | FMS 상세제원 연계 추가 수신 or 자체입력 |

### 8-1. 결론

- FMS BASTB_MASTER 42컬럼으로 충족되는 위험성평가 변수는 **6종(정적 배경변수)뿐**이다: 경과연수·종별·시설물구분·관리주체코드·주소·내진적용여부.
- 위험도 산정의 **핵심 동적변수 4축**(안전등급·점검이력·중대결함·규모)은 전부 없다. 이 4축 없이는 위험성평가를 근거 있게 시작할 수 없다.
- 규모 필드 부재로 **중대재해법 별표3상 '공중이용시설 해당 여부' 자동판정 자체가 불가**하다 — base-targets '공중이용시설(47)' 산정근거는 규모 확보 후 재검증해야 한다.

### 8-2. 즉시 착수 가능한 최소 정보집합 (MVP)

FMS 추가 연계를 기다리지 않고 프로토타입에서 위험성평가를 굴리려면, 아래 4축을 **보완입력(자체 소유)**으로 확보한다:

1. 안전등급(A~E) — 드롭다운, 시설물안전법 별표8 정의 툴팁
2. 최근 점검일 + 차기 점검예정일 — 등급·종별로 법정주기 자동 제시 후 수정
3. 중대결함 유무 — 있으면 유형·통보일·조치기한
4. 규모 대표값 1개(교량=연장/경간, 건축물=연면적, 하수=처리용량) + 이용인원

이 4축 + FMS 보유 6종이면 FAC03-V 위험도 스코어링과 위험성평가 착수가 성립한다.

## 9. 후속 조치

### 9-1. FMS 추가 연계 요청(질의서 2차)
- 시설물관리대장 **상세제원(BASTB_DTL_*)** 연계 — 규모·상부구조형식 확보
- **점검진단실적** 연계 — 안전등급·점검이력 확보
- **중대결함사후관리** 수신 — 결함 원천 확보
- 수신(receive) 인터페이스 reqMethod·자료구조 협의(가이드 미명시)

### 9-2. 코드 정의 확인(가이드 stale)
- FACIL_CLASS `2/3` 값 정의(가이드는 `1,2,9`)
- DSN_KEEP_STATUS `1/2/3/4` 값 정의(가이드는 `Y,N`)
- FACIL_GBN/KIND 현행 공통코드 → 국토안전관리원 FMS 최신 공통코드 다운로드
- BASTB_DTL_* suffix 결정 필드(현행 코드 기준)

### 9-3. 자체 확보
- 좌표 지오코딩(80건), 소관부서·담당자 매핑, 이용인원/통과교통량 출처 확정, 소관 미상 31건(공란 21 + 국가/민간 10) 정리

- [ ] FAC01-V가 FMS 80건을 `FACIL_NO` 기준으로 목록화하고 종별·구분·소관·노후·결측 필터가 동작한다
- [ ] FAC02-D가 42컬럼을 읽기전용으로, 보완입력 4축 이상을 편집 가능으로 분리 표시한다
- [ ] FMS 수신이 자체 소유 필드(보완입력)를 덮어쓰지 않는다
- [ ] FAC04-S 수신이 차이 미리보기 → 선택 반영 단계를 거치며 적층 모달을 만들지 않는다(단일 모달 규칙)

- [ ] `FACIL_NO`가 위험성평가 대상(risk-form)으로 전달돼 대상이 자동 선택된다
- [ ] 위험도 산정 시 핵심 변수 미입력은 '산정불가'로 분류하고 0으로 처리하지 않는다
- [ ] 소관 미상·좌표 결측·코드 sentinel이 태그/안내로 노출된다
- [ ] 첨부 안내는 `DYV2.fileHint()`, 조직도는 `DYV2.ORG`를 재사용한다(CLAUDE.md §2·§3)
- [ ] admin-integration.html(SIR-001)을 수정하지 않는다

## 11. 검토했으나 갭이 아닌 것

- **'FMS 동기화로 연면적을 채운다'** — 불가. 연면적은 BASTB_MASTER에 없고 상세제원 소관이며 이번 파일에 0건. base-targets가 표시하던 연면적은 앱 내부 하드코딩값이었다.
- **'BASTB_MASTER의 관리주체코드로 소관부서를 안다'** — 불가. MNG_MAIN_CD는 관리주체 코드일 뿐 앱 조직도 부서와 정합되지 않는다. 소관부서는 자체 매핑 필요.
- **'80건이 곧 공중이용시설 47건'** — 미검증. 엔티티가 거의 겹치지 않으며(건축물 22건 vs 용도분류 47건), 규모 판정 전에는 확정 불가.
