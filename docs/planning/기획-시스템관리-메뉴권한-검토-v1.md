## 0. 검토 대상 기획 요약 (발주 초안)

**메뉴 관리**
- 좌측 트리(대메뉴/중메뉴)에서 메뉴 선택 → 우측 설정 패널
- 접근 부여 방식 2가지를 **라디오**로 선택, 선택에 따라 속성창 전환
  - 방식 1: 권한 관리에서 세팅한 **권한(등급)을 불러와 적용** — 선택한 등급의 구성원(부서·사용자)과 권한 범위를 함께 표시
  - 방식 2: **조직도 팝업으로 부서·사용자를 직접 지정** — 항목별 체크박스로 보기/수정 선택

**권한 관리**
- 권한 등급을 사용자가 추가·제거
- 기본 5등급(안): 군수 / 부군수·국장 / 과장·팀장 / 담당부서 / 일반 공무원
- 등급별로 조직도 팝업으로 부서·개인을 지정, 지정 목록에 읽기/수정/전체 체크박스 (전체 체크 시 읽기·수정 비활성)

## 1. 총평

**구조 방향은 타당하다.** "권한 관리 = 누구인가(등급 정의·구성원)" / "메뉴 관리 = 어디에 접근하나(메뉴×권한 매핑)"의 2계층 모델은 행정정보시스템의 표준 패턴이며, `_공통_권한정의.md`의 화면군×역할 매트릭스(§5)와 화면별 매핑(§6)을 그대로 담아낼 수 있는 그릇이다. 이 기획이 확정되면 공통 권한정의서의 미결사항 **PERM-001(역할-직위 매핑)·PERM-007(역할 부여 방식)·PERM-008(접근제어 범위)** 이 해소된다.

다만 확정 전에 반드시 정리해야 할 **구조 쟁점 3건**(§2)과 **보완 필요사항 다수**(§3~§5)가 있다. 특히 쟁점 1(읽기/수정 체크박스의 위치)은 현안대로 가면 화면정의서의 권한 요구를 담지 못하므로 우선 결정이 필요하다.

UI/UX 실현성: **충분히 가능** (§6). 두 화면 모두 좌 목록 + 우 상세의 마스터-디테일 골격을 공유하고, 조직도 선택은 페이지 레벨 단일 모달 1개로 처리되어 단일 모달 규칙과 충돌하지 않는다.

## 2. 구조 쟁점 3건 (확정 전 필수 결정)

### 쟁점 1 — 읽기/수정 체크박스가 "등급 구성원"에 붙는 구조의 문제 ★최우선

현안은 권한 관리에서 **등급 안의 구성원마다** 읽기/수정/전체를 체크한다. 이 구조의 문제:

1. **등급의 의미가 무너진다.** 같은 등급 안에서 구성원마다 권한이 다르면 "이 등급은 무엇을 할 수 있는 등급인가"에 답할 수 없다. 메뉴 관리 방식 1에서 등급을 불러와 적용할 때, 그 등급이 해당 메뉴에서 갖는 권한 수준이 구성원마다 달라져 예측 불가능해진다.
2. **메뉴별 차등이 불가능하다.** 권한 수준이 등급(구성원)에 전역으로 붙으면 "일반 공무원은 대부분 조회만, 단 의견청취·건의함에서는 등록 가능" 같은 요구를 표현할 수 없다. 이런 요구는 실제로 존재한다 — 의견청취는 전 직원 제출이 전제인 기능이다.
3. **화면정의서 체계와 어긋난다.** `_공통_권한정의.md` §5(화면군 × 역할)·§6(화면 × 역할)은 권한 수준을 **화면과 역할의 교차점**에 둔다. 즉 기존 정의서 체계 자체가 "권한 수준은 메뉴별로 다르다"를 전제한다. 이 발판이 정의서의 권한 내용을 포괄하려면 같은 구조여야 한다.

**대안 비교**

| 안 | 구조 | 평가 |
|---|---|---|
| A안(현안) | 등급 구성원별 읽기/수정/전체 (전역) | 메뉴별 차등 불가 · 등급 의미 약화 → 비권장 |
| **B안(권장)** | 권한 관리 = 구성원만 관리(체크박스 없음) / 보기·수정 체크는 **메뉴 관리의 지정 행**에서 설정 | 메뉴별 차등 가능 · "누구"와 "무엇"의 분리가 명확 · 정의서 §5·§6과 동형 |
| C안 | 등급에 기본 수준 + 메뉴별 오버라이드 | 유연하지만 "실제 적용값이 어디서 왔는지" 혼동 위험, 프로토타입에 과함 |

**B안 채택 시 조정 내용**
- 권한 관리의 구성원 행: 체크박스 제거, [제거] 버튼만. 등급 속성으로 **'전체 권한' 플래그**만 유지(아래 참조).
- 메뉴 관리 방식 1: 등급 선택 시 지정 행이 추가되고 **그 행에 보기/수정 체크**. 행을 펼치면 등급 구성원(부서·인원) 미리보기 — 발주 요구("어떤 부서·사용자고 어디까지인지 보여줘")는 이 미리보기로 충족.
- **'전체 권한'의 재정의**: 구성원별 체크가 아니라 **등급 속성**으로 격상. 전체 권한 등급(시스템 관리자 전용)은 메뉴별 설정과 무관하게 모든 메뉴 보기+수정. 메뉴 관리 화면에는 "전체 권한 등급 N개는 자동 적용됩니다" 안내만 노출하고 개별 설정 대상에서 제외 → 관리자가 실수로 자기 접근을 제거하는 사고(셀프 잠금)를 원천 차단.

### 쟁점 2 — 라디오 "둘 중 하나"의 배타 구조는 실사용과 충돌

하나의 메뉴는 통상 **등급 기반 + 개별 지정이 동시에** 필요하다. 실제 데이터가 이를 증명한다 — `admin-users.html`의 기존 세팅:

| 사용자 | 부서 | 기존 등급 | 담당 대메뉴 |
|---|---|---|---|
| 이보건 | 보건소 | 부서 담당 | 안전보건교육 |
| 정환경 | 환경과 | 부서 담당 | 유해·위험요인 관리 |
| 최회계 | 회계과 | 부서 담당 | 도급관리 |

'안전보건교육' 메뉴 하나만 봐도: 전담부서(등급) 수정 + 보건소 담당자(개별) 수정 + 상위 등급들 보기가 **동시에** 걸려야 한다. 라디오가 메뉴 단위로 방식을 배타 선택하게 하면 이 구성이 불가능하다.

**권장** — 라디오는 유지하되 의미를 바꾼다: "이 메뉴의 방식 선택"이 아니라 **"추가 방식 전환"**(입력 수단 토글)으로. 라디오 아래 속성창(등급 선택기 ↔ 조직도 지정기)이 전환되는 UI는 발주안 그대로 두고, **적용 결과는 하나의 통합 목록**(유형 배지: `권한등급`/`부서`/`사용자`)에 누적한다.
- 동일 대상에 중복 경로로 권한이 걸리면(등급 경유 보기 + 개별 지정 수정) **높은 권한 우선(합집합)** 을 명문화한다.

### 쟁점 3 — 기본 5등급 검수 결과 (발주 질의 사항)

발주안: 군수 / 부군수·국장 / 과장·팀장 / 담당부서 / 일반 공무원 — **방향은 타당하나 4가지 보정 필요.**

**(1) '시스템 관리자' 누락 — 필수 추가.** 메뉴 관리·권한 관리 자체에 접근할 등급이 발주안에 없다. `_공통_권한정의.md` §5는 "시스템 관리 화면군 = 시스템 관리자만 O"로 못 박고 있고, 현재 시스템 관리자는 재난안전과 중대재해팀(박안전)이다. 이 등급이 없으면 권한 체계가 스스로를 관리하지 못한다(잠금 사고). **삭제 불가·구성원 0명 불가**의 기본 제공 등급으로 추가해야 한다.

**(2) '담당부서' 명칭의 중의성 — 개명 권장.** 이 시스템에서 '담당부서'는 이미 **문서·업무별 소관 부서**를 뜻하는 컬럼명으로 전면 사용 중이다(업무 목록의 담당자·담당부서 열 등). 등급명으로 재사용하면 두 개념이 충돌한다.
- 발주 의도가 **전담조직(재난안전과 중대재해팀)** 이라면 → 등급명을 **'전담부서(중대재해 총괄)'** 등으로 개명.
- 발주 의도가 "각 업무의 담당부서"라면 → 이는 **메뉴마다 구성원이 달라지므로 고정 구성원을 갖는 등급으로 성립 불가.** 그 케이스는 등급이 아니라 메뉴 관리의 **개별 지정**으로 처리해야 한다(아래 3).

**(3) '부서별 업무 담당자' 계층의 처리 방식 — 등급이 아닌 개별 지정으로.** 이 시스템의 최다 실사용자는 각 부서에서 문서 작성·점검표 제출을 수행하는 담당 주무관(직급상 '일반 공무원')이다. 이들을 일반 공무원 등급(조회)에 두면 업무 수행이 불가능하다. 그렇다고 '부서 담당' 전역 등급을 만들면 — 등급 구성원은 메뉴를 가리지 않으므로 — 회계과 담당자가 교육 메뉴까지 수정하게 되는 부작용이 생긴다. **결론: 부서별 담당자는 메뉴 관리의 개별 지정(방식 2)으로 해당 메뉴에만 수정 권한을 부여하는 것이 원칙.** 이것이 방식 2의 존재 이유이며, 쟁점 2(병존)가 전제 조건이 된다.

**(4) '일반 공무원' = 수동 지정 불가 규모 — 자동 구성원 플래그 필요.** 전 직원(수백 명)을 조직도에서 일일이 담는 것은 비현실적이다. 등급 속성에 **'구성원 자동(전 직원)'** 옵션을 두어, 이 등급은 구성원 지정 UI 대신 "SSO 로그인 사용자 전원 자동 포함"으로 표시한다. (admin-users의 SSO·조직도 자동 동기화 전제와 일치)

**권장 기본 등급 6종**

| # | 등급 | 구성원 | 비고 |
|---|---|---|---|
| 1 | 시스템 관리자 | 전담팀 관리자(개인) | 전체 권한 플래그 · 삭제 불가 · 0명 불가 |
| 2 | 군수 | 개인 1 | 경영책임자(중대재해처벌법상) — 초기 프리셋: 전 메뉴 보기 |
| 3 | 부군수·국장 | 개인 소수 | 초기 프리셋: 전 메뉴 보기 |
| 4 | 과장·팀장 | 직책자 개인 지정 | 부서 단위로 넣으면 부서원 전원이 포함되므로 **개인 지정 원칙** 운용 규칙 명시 |
| 5 | 전담부서(중대재해 총괄) | 재난안전과 중대재해팀(부서) | 발주안 '담당부서' 개명 — 업무 메뉴 전반 수정 |
| 6 | 일반 공무원 | 자동(전 직원) | 대시보드·통계 등 보기 + 의견청취 등록 |

**기존 체계와의 매핑** (기획서에 포함 권장)

| 기존 | 새 체계 |
|---|---|
| 정의서 4역할: 시스템 관리자 | 시스템 관리자 |
| 정의서: 담당자 / admin-users: 총괄 담당 | 전담부서(중대재해 총괄) |
| admin-users: 부서 담당 | (등급 아님) 메뉴별 개별 지정 |
| 정의서: 부서장 | 과장·팀장 |
| 정의서: 열람자 / admin-users: 열람 | 일반 공무원 |
| (없음) | 군수 · 부군수·국장 — 상위 열람 계층 신설 |

> 군수·부군수·국장의 분리 유지 여부: 메뉴 접근 관점의 실효는 동일(전 메뉴 보기)할 가능성이 높으나, 군수는 경영책임자로서 별도 표기의 상징·보고 가치가 있어 분리 유지가 타당하다. 결재 권한은 온나라 연계로 외부화되어 있어(결재선 '팀장→과장→부군수' 하드코딩 → 온나라 수신 구조) 시스템 등급과 무관하다.

## 3. 메뉴 관리 화면 — 보완 필요사항

| # | 항목 | 내용 · 권장 |
|---|---|---|
| M-1 | **미설정 메뉴 기본 정책** | 아무 권한도 설정하지 않은 메뉴의 취급이 미정의. 권장: **기본 차단**(시스템 관리자 제외) + 시스템 오픈 시 `_공통_권한정의.md` §5 매트릭스 기반 **초기 세팅 프리셋** 일괄 제공. 트리 노드에 `미설정` 배지로 가시화 |
| M-2 | **접근 불가 메뉴의 GNB 처리** | 숨김 vs 비활성 미정의. 권한정의서 표기 규칙(숨김/비활성 구분)과 맞춰 **숨김** 권장. 대메뉴는 "하위 중메뉴 1개 이상 접근 가능 시 노출" 규칙 명시 |
| M-3 | **설정 단위 = 중메뉴** | 권한 설정은 중메뉴(화면) 단위로만. 대메뉴 노드 선택 시 우측은 **하위 중메뉴 권한 요약표 + [하위 일괄 적용]** — 대메뉴에 직접 설정을 허용하면 상속·오버라이드 규칙이 필요해져 복잡도만 증가 |
| M-4 | **저장 방식·이탈 보호** | 미정의. 권장: 메뉴별 명시 [저장]/[초기화] 버튼(권한 변경은 파급이 커 자동저장 부적합) + 미저장 상태에서 트리의 다른 노드 클릭 시 확인 다이얼로그 |
| M-5 | **기본 정보 영역** | 패널 상단에 메뉴 기본 정보(메뉴명·경로·연결 화면 SFR/SCR 코드·설명) 조회 + **사용 여부 토글**(메뉴 전체 숨김). 메뉴 추가·삭제·순서 변경은 범위 외(메뉴 = 개발 산출물)임을 기획서에 명시 |
| M-6 | **적용 요약** | 설정 하단에 "권한등급 n · 부서 n · 개인 n명" 요약 — 설정 결과를 한눈에 확인 |
| M-7 | **설정 복사** | 중메뉴 29개를 개별 세팅하는 노동 절감: [다른 메뉴 설정 불러오기(복사)] 제공 권장 |

## 4. 권한 관리 화면 — 보완 필요사항

| # | 항목 | 내용 · 권장 |
|---|---|---|
| P-1 | **수정 ⊃ 읽기 규칙** | (메뉴 관리의 지정 행 체크박스에도 동일 적용) 수정 체크 시 보기 자동 체크·잠금. "수정 가능한데 조회 불가"의 모순 차단 |
| P-2 | **등급 삭제 가드** | 등급이 메뉴에 적용 중이면 "N개 메뉴에 적용 중 — 삭제 시 해당 메뉴에서 지정이 제거됩니다" 경고 확인. 등급 목록에 참조 수 배지 |
| P-3 | **기본 등급 보호** | 시스템 관리자 등급: 삭제 불가 + 구성원 최소 1명 유지(마지막 구성원 제거 차단) — 셀프 잠금 방지 |
| P-4 | **중복 감지** | 같은 등급에 부서 + 그 부서 소속 개인을 중복 추가 시 안내. 한 사용자가 여러 등급에 속하는 겸직(PERM-003)은 허용하되 실효 권한 = 합집합(최대) 명문화 |
| P-5 | **부서 지정의 의미** | 시점 스냅숏이 아닌 **부서 단위 동적 포함**(전입자 자동 반영 — SSO 조직도 동기화 전제). 인사이동 시 개인 지정은 잔존하므로 사용자 관리 화면에서 정리하는 운용 흐름 명시 |
| P-6 | **하위 부서 포함 옵션** | 실제 조직도는 실·국 > 과 > 팀 계층 — 부서 선택 시 '하위 부서 포함' 체크 옵션 |
| P-7 | **변경 이력** | 공공 시스템 감사 요건: 권한 변경 이력(누가·언제·무엇을) — PRD 요구사항으로 기록, 프로토타입은 목업 수준 |

## 5. 범위 경계 (기획서·PRD에 명시할 문장)

1. **메뉴 권한 vs 기능 단위 권한** — 이 기능은 메뉴의 **접근(보기)·편집(수정) 게이트**까지만 다룬다. '수정'은 등록·수정·삭제·상신을 포함하며, 승인·반려·공표 등 **결재 단계 권한은 각 화면 정책(화면정의서 §2)과 온나라 결재선**의 영역으로 존치한다. (PERM-004·005의 부분 해소)
2. **메뉴 권한 vs 데이터 스코프** — "본인 담당만/소속 부서만/전체" 같은 **데이터 노출 범위(PERM-002)는 이번 범위에서 제외**하고 각 화면 정책으로 남긴다. (지정 행에 '조회 범위' 셀렉트를 추가하는 확장은 Phase 2 후보)
3. **프로토타입 표현 한계** — 프로토타입에는 로그인·접근 강제가 없으므로(정의서 §4), 두 화면은 설정 UI·상호작용의 목업이며 실제 차단은 구현 목표로 PRD에 기술한다.
4. **연계 문서 개정** — 이 기획 확정 시 `_공통_권한정의.md`를 v2로 개정한다: §3 역할 정의 → 새 6등급 체계, §5 매트릭스 → 초기 세팅 프리셋으로 재규정, §7 PERM-001·007·008 해소 처리. 화면정의서 8종의 §2가 이 문서를 참조하므로 일관성 유지 필수.
5. **기존 화면 재편** — 시스템 관리 하위: '사용자·권한'(SFR-015)은 **'사용자 관리'로 존치**(SSO 동기화 + 사용자별 실효 권한 역조회)하고 '메뉴 관리'·'권한 관리'를 신설한다. 사용자 목록의 '권한'·'담당 대메뉴' 컬럼은 새 체계의 산출값(읽기 전용)으로 재정의. 화면 코드는 기존 관례에 따라 SCR-ADMIN-005(메뉴 관리)·SCR-ADMIN-006(권한 관리), 메뉴 코드 ADM01-S·ADM02-S 제안.

## 6. UI/UX 실현성 평가 — 판정: 가능

**공통 골격** — 두 화면 모두 `좌측 목록(트리/등급 리스트) + 우측 상세 패널`의 마스터-디테일. 같은 골격을 공유하므로 학습 비용이 낮고, 기존 프로토타입 자산(카드·테이블·칩·인라인 조직도 트리)으로 구성 가능하다.

**메뉴 관리 화면 구성(안)**
```
┌ 좌: 메뉴 트리 ────────┐ ┌ 우: 설정 패널 ──────────────────────┐
│ 검색                  │ │ [기본 정보] 메뉴명·경로·연결 화면 코드  │
│ ▾ 대시보드            │ │            사용 여부 토글             │
│ ▾ 안전보건관리체계     │ │ [접근 권한]                          │
│    · 경영방침 ●설정됨  │ │  ◉ 권한 불러오기  ○ 직접 지정 ← 라디오 │
│    · 위험성평가 ○미설정│ │  (전환되는 추가 속성창)               │
│ ▸ 의견청취            │ │  ─ 적용 목록(통합) ─                  │
│ …9개 대메뉴/29개 중메뉴│ │  [등급] 전담부서   ☑보기 ☑수정 [제거] │
│                       │ │    └ 구성원 미리보기(펼침)            │
│                       │ │  [부서] 보건소     ☑보기 ☑수정 [제거] │
│                       │ │  [사용자] 김안전   ☑보기 ☐수정 [제거] │
│                       │ │  요약: 등급1·부서1·개인1 [저장][초기화]│
└───────────────────────┘ └──────────────────────────────────────┘
```

**권한 관리 화면 구성(안)**
```
┌ 좌: 등급 목록 ─────────┐ ┌ 우: 등급 상세 ───────────────────────┐
│ [+ 등급 추가]          │ │ [기본 정보] 등급명·설명               │
│ 시스템 관리자 🔒(1명)   │ │   전체 권한 ☐ / 구성원 자동(전직원) ☐ │
│ 군수 (1명)             │ │ [구성원] [+ 조직도에서 추가]          │
│ 부군수·국장 (4명)       │ │  [부서] 재난안전과 중대재해팀 [제거]   │
│ 과장·팀장 (23명)        │ │  [사용자] 박안전 (재난안전과) [제거]   │
│ 전담부서 (부서1)        │ │ [적용 현황] 이 등급이 적용된 메뉴 N개  │
│ 일반 공무원 (자동)      │ │                        [저장][초기화] │
└───────────────────────┘ └──────────────────────────────────────┘
```

**조직도 선택기** — 두 화면 공용. 페이지 레벨에서 여는 **단일 모달 1개**이므로 단일 모달 규칙(CLAUDE.md)과 충돌 없음. 단, 기존 조직도 컴포넌트는 단일 선택용이므로 **복수 선택형으로 확장 설계** 필요: 좌 트리(부서·개인 체크) + 우 선택 바스켓, 검색, '하위 부서 포함' 체크, [적용]. 등급 추가·수정 폼은 모달이 아닌 우측 패널 인라인으로 두어(패널 안에서 조직도 모달을 열어도 모달은 항상 1개) 적층을 원천 방지한다.

**사용성 보강 6건(권장)** — ① 트리 노드 설정 상태 배지 ② 적용 요약 행 ③ 다른 메뉴 설정 복사 ④ 대메뉴 하위 일괄 적용 ⑤ 초기 세팅 프리셋 ⑥ 사용자 기준 역조회("이 사용자가 접근 가능한 메뉴는?" — 사용자 관리 화면에 배치)
