# 기획 — 업무 목록 / 이행 목록 「업무 등록」 기능 상세 설계 (CRUD) v1

> 대상 화면: `docs-preset.html`(업무 목록) · `docs-exec.html`(이행 목록)
> 작성 관점: 개발 착수용 상세 기획 — UI·데이터·분기·CRUD·연계를 한 문서에 정리
> 분류 정본: **법적의무 재분류 v2**(9 대메뉴 · 53 세트, `js/sets-data-v2.js`) 기준
> 미확정 항목은 §13 미결사항에 모았습니다.

---

## 1. 목적 · 배경

현재 업무 목록·이행 목록은 **조회 전용**이다. 문서를 새로 만들거나(등록), 메타를 고치거나(수정), 빼는(삭제) 기능이 없어, 모든 문서가 `js/sets-data-v2.js`에 하드코딩된 시드 데이터에만 의존한다.

본 기획은 두 화면에서 **하나의 등록 마법사(wizard)** 로 문서를 추가하고, 등록한 문서를 수정·삭제까지 할 수 있는 **완전한 CRUD**를 정의한다. 등록 시 사용자는 다음을 순서대로 결정한다.

1. **어느 법령 분류(대메뉴)·세트·PDCA 단계**에 속하는 문서인가
2. **처리유형**(첨부파일 / 전자문서 / 프로그램)이 무엇인가
3. 처리유형에 따라 **달라지는 하위 입력 폼**을 채운다
   - 전자문서 → **전자문서 빌더의 양식(템플릿)과 연계**해 어떤 양식으로 입력받을지 선택
   - 프로그램 → **어느 메뉴 화면으로 연결**할지 선택

## 2. 범위 (Scope)

### 2-1. 포함 (CRUD)

| 기능 | 설명 | 진입점 |
|---|---|---|
| **C 등록(Create)** | 마법사로 새 업무문서 1건 생성 | 업무 목록·이행 목록 툴바 [+ 업무 등록] |
| **R 조회(Read)** | (기존) 목록·문서 상세에서 시드+사용자 문서를 함께 표시 | 목록 카드 / `doc-detail.html` |
| **U 수정(Update)** | 등록한 문서의 분류·처리유형·폼 값을 다시 편집 | 문서 상세 [편집] · 목록 행 [⋯ → 편집] |
| **D 삭제(Delete)** | 등록한 문서를 삭제(확인 후) | 문서 상세 [삭제] · 목록 행 [⋯ → 삭제] |

### 2-2. 제외 (이번 범위 아님)

- 시드 문서(`DY_DOCS_V2` 원본 429건)의 **물리 삭제** — 사용자 편집/삭제는 오버레이로 처리(§6-4), 원본은 보존.
- 전자문서 빌더(`admin-presets.html`) 자체의 양식 CRUD — 본 기획은 **빌더 양식을 선택·연계**하는 것까지만.
- 실제 결재(온나라)·파일 스토리지 연동 — 기존과 동일하게 프로토타입 시뮬레이션(toast/모달).
- 권한별 등록 제한 — §13 미결.

## 3. 핵심 개념 — 등록 시 결정하는 값

문서 1건은 아래 축으로 분류된다. 마법사는 이 값을 순서대로 받는다.

| 축 | 의미 | 입력 UI | 값 출처 |
|---|---|---|---|
| **① 법령 분류(대메뉴)** | 9개 법적의무 분류 중 하나 | **탭 선택형** | `DY_MENUS_V2` / `MENU_ORDER_V2` |
| **② 세트(업무 묶음)** | 대메뉴 하위 PDCA 세트(A1·B1…) | 드롭다운(①로 필터) | `DY_SETS_V2` |
| **③ PDCA 단계** | P 계획 / D 실행 / C 점검 / A 조치 | **라디오** | 고정 4값 |
| **④ 처리유형** | 첨부파일 / 전자문서 / 프로그램 | 라디오(카드형) | 고정 3값 |
| **⑤ 처리유형별 상세** | ④에 따라 폼이 분기 | §5 | — |

> **법령 분류 = 9 대메뉴**: v2 분류는 "법적의무 재분류"이며 9개 대메뉴가 곧 법령 분류축이다. 세트(②)는 그 하위의 업무 묶음으로, 등록 문서를 목록의 어느 세트 카드에 넣을지 결정한다.

### 9 대메뉴 (탭 순서 = `MENU_ORDER_V2`)

`안전보건경영(safety)` · `위험성평가(risk)` · `관계법령(law)` · `의견청취(opinion)` · `교육(edu)` · `도급관리(contract)` · `발주공사(order)` · `비상조치(emergency)` · `재해대응(incident)`

## 4. 화면 흐름 — 등록 마법사 (4 STEP)

### 4-0. 형태 · 위치

- **컴포넌트**: 신규 `js/doc-register.js`, 전역 `window.DYREG`.
- **렌더 방식**: 기존 `EDOC`와 동일한 2-모드.
  - `DYREG.openCreate(opts)` — **모달 마법사**(등록). 목록 툴바 버튼에서 호출.
  - `DYREG.openEdit(docId, opts)` — 같은 마법사를 **기존 값으로 채워** 호출(수정).
- 모달 1개 안에서 **STEP 1→2→3→4**를 [다음]/[이전]으로 이동(스텝 인디케이터 상단 고정). 페이지 이동 없음.
- 진입 버튼: 두 화면 툴바의 `목록 다운로드` 옆에 **`+ 업무 등록`**(`btn btn-primary btn-sm`) 추가.

### 4-1. STEP 1 — 분류 (어디에 속하는 문서인가)

| 영역 | 구성 | 동작 |
|---|---|---|
| 법령 분류(대메뉴) | **탭 9개** (`safety`…`incident`) | 탭 클릭 → 선택 표시 + 아래 세트 목록 갱신 |
| 세트 | 드롭다운 — 선택 대메뉴의 세트(`DY_SETS_V2.filter(menuKey)`) + `세트 미지정` 옵션 | 세트 선택 시 그 세트의 `pdcaFlow`를 힌트로 표시(예: "이 세트 단계 구성: P·D·C·A") |
| PDCA 단계 | **라디오 4개**: `P 계획` `D 실행` `C 점검` `A 조치` | 단일 선택. 세트의 `pdcaFlow`에 없는 단계는 선택 시 안내(경고만, 차단 안 함) |

- **검증(다음 이동 조건)**: 대메뉴 필수, PDCA 필수. 세트는 선택 권장(미선택 시 `세트 미지정` 그룹으로 들어감).

### 4-2. STEP 2 — 처리유형 선택

- **라디오 카드 3종** (아이콘 + 제목 + 한 줄 설명):

| 값 | 라벨 | 설명(카드 보조문구) | 상세 화면에서의 동작(기존 규칙) |
|---|---|---|---|
| `첨부파일` | 📄 첨부파일 | 한글·PDF 등 완성 문서를 올려 보관 | 상세에서 **미리보기/다운로드** |
| `전자문서` | 🧾 전자문서 | 시스템 양식에 값을 입력해 작성 | 상세에서 **입력 폼** |
| `프로그램` | 🖥️ 프로그램 | 전용 화면(위험성평가 등)에서 처리 | 상세에서 **해당 화면으로 이동** |

- 선택 즉시 STEP 3의 폼 구성이 결정된다(§5).

### 4-3. STEP 3 — 상세 입력 (처리유형별로 폼 전체가 분기) — 2026-06-23 개정

STEP 3은 **처리유형에 따라 입력 항목이 다르다.** 단, **담당(담당자) 지정은 3종 공통**이다. 처리유형별 상세는 §5 참조.

**담당 블록 (3종 공통)** — `ownerBlock()`
- 담당부서: 읽기전용 표시(**클릭 불가** — 자동 표시 전용).
- **[담당자 지정] 버튼(활성)** → 조직도 팝업에서 한 명 선택 시 담당부서+담당자 동시 설정. 버튼 라벨이 선택한 담당자로 갱신("담당자: ○○○ (변경)"). 별도 담당자 입력칸은 두지 않는다.
- 담당자는 3종 모두 **필수**.

| 처리유형 | STEP 3 입력 항목 |
|---|---|
| 첨부파일 | 문서명 · **담당(공통)** · 주기 · 법적 근거 · **파일(추가·목록·삭제·보존연한)** |
| 전자문서 | 문서명 · **담당(공통)** · 연계 양식 (그 외 항목은 전자문서 양식 안에서 입력) |
| 프로그램 | 연결 메뉴 → 연결 뎁스(화면) · **담당(공통)** (문서명은 화면 이름으로 자동) |

### 4-4. STEP 4 — 확인 · 저장

- **요약 카드**: 대메뉴·세트·PDCA·처리유형·문서명·연계 대상(양식/프로그램)·담당을 한눈에.
- 버튼: [이전] · [저장]. 저장 시 §6·§7 규칙으로 영속 → 모달 닫고 목록 새로고침 + `toast('등록되었습니다')`.
- (수정 모드일 때) 버튼 라벨 [저장] → 동일, 추가로 [삭제] 노출.

## 5. 처리유형별 분기 폼 (STEP 3 하위) — 가장 중요한 분기

### 5-A. 첨부파일 선택 시

| 필드 | 키 | 유형 | 필수 | 비고 |
|---|---|---|---|---|
| 문서명 | `name` | text | ✅ | |
| 담당 | `dept`/`assignee` | **담당 블록(공통)** | ✅ | §4-3 |
| 주기 | `cycle` | select: 연간/반기/분기/월/수시/발생시/상시 | | |
| 법적 근거 | `legalBasis` | text(직접 입력) | | |
| 파일 추가 | `files[]` | 파일명+설명 입력 후 **[파일 추가]** → 목록 누적 | | 다중, 행별 [삭제] |
| 첨부파일 목록 | — | 추가된 파일 테이블(파일명·설명·삭제) | | `DYREG._addFile`/`_delFile` |
| 보존 연한 | `retention` | select: 3년/5년/10년/영구 | | |

- **파일 첨부 후 목록 표시**: 파일명+설명을 입력해 [파일 추가]하면 아래 목록 테이블에 누적되고 행별 삭제 가능. 문서 상세(`previewBody`)에도 동일 목록을 표시.

### 5-B. 전자문서 선택 시 — **문서명 + 전자문서 빌더 양식 연계만**

| 필드 | 키 | 유형 | 필수 | 비고 |
|---|---|---|---|---|
| 문서명 | `name` | text | ✅ | |
| 연계 양식 | `formRef` | **양식 선택 드롭다운** | ✅ | 아래 양식 소스 |
| 양식 미리보기 | — | 읽기전용 미리보기 | | 선택 양식의 필드 레이블 나열 |
| 새 양식 만들기 | — | 링크 → `admin-presets.html` | | 빌더로 이동(새 탭) |

- 담당·주기·법적근거 등 나머지 항목은 **전자문서 양식 안에 포함**되므로 STEP 3에서 받지 않는다.

**양식 소스(연계 대상)** — 두 계층을 합쳐 보여준다:

1. **표준 폼 7종** (`EDOC_T.FORMS` F1~F7): 계획서·실시결과·회의록·교육결과·점검표·평가서·조치보고
2. **빌더 프리셋 양식** (`admin-presets.html`의 사용자 정의 양식): 계획·실적 공통, 회의록 양식 등

- 선택값은 `{ kind: 'standard'|'preset', formId: 'F1'|presetId }`로 저장.
- 저장 시 상세(`doc-detail`)는 기존 `formBody()` + `EDOC.renderInline(form: 선택양식)`로 입력 폼 렌더 → 작성중→등록완료→확정(온나라) 상태머신 그대로 동작.
- **PDCA 단계 자동 추천**: STEP 1의 PDCA / 문서명으로 `EDOC_T.formForDoc()` 휴리스틱을 돌려 양식을 **기본 선택**해 주고, 사용자가 바꿀 수 있게 한다(예: 점검 단계 → F5 점검표 기본 선택).

### 5-C. 프로그램 선택 시 — **어떤 메뉴의 어떤 뎁스로 넘길지만 선택**

| 필드 | 키 | 유형 | 필수 | 비고 |
|---|---|---|---|---|
| 연결 메뉴 | (nav) | **드롭다운(GNB 메뉴)** | ✅ | `NAV_TREE` 상위 |
| 연결 뎁스(화면) | `linkHref` | **드롭다운(메뉴 하위 화면)** | ✅ | 메뉴 선택 시 갱신 |

- 문서명·담당 등 별도 입력 없음 — **문서명은 선택한 화면 이름으로 자동 설정**(`linkLabel`).
- 저장 모델: `linkHref`(이동 URL) · `linkLabel`(화면명) · `linkMenu`(상위 메뉴명).

**연결 가능 메뉴 → 뎁스** (`NAV_TREE`):

| 메뉴(GNB) | 뎁스(화면) → href |
|---|---|
| 대시보드 | 대시보드 → `index.html` |
| 기본정보 | 평가대상 관리 → `base-targets.html` · 일괄 등록 → `base-bulk.html` |
| 안전보건관리체계 | 위험성평가 → `risk-list.html` · 유해·위험요인 관리 → `proc-list.html` · 경영방침 → `menu.html?m=policy` · 조직 → `menu.html?m=org` · 안전보건교육 → `edu.html` · 의견청취 → `menu.html?m=opinion` · 도급관리 → `menu.html?m=contract` · 개선조치 → `menu.html?m=improve` · 이행관리 → `menu.html?m=comply` |
| 업무문서 | 기준문서함 → `docs-archive.html` · 업무 목록 → `docs-preset.html` · 이행 목록 → `docs-exec.html` |
| 통계·보고 | 통계 → `stats.html` · 보고서 → `reports.html` · 정보센터 → `info-center.html` |
| 시스템 관리 | 사용자 관리 → `admin-users.html` · 알림 설정 → `admin-notify.html` · 프리셋 양식 관리 → `admin-presets.html` · 연계 관리 → `admin-integration.html` |

- 저장 시 상세(`doc-detail`)는 `programBody()`로 "○○ 화면으로 이동 →" 버튼(href=`linkHref`) + "상위메뉴 › 뎁스" 안내 렌더.

> **분기 요약**: STEP 2의 선택값 하나가 STEP 3 하위 폼 + 문서 상세 렌더 + 목록 액션 라벨(입력/미리보기/이동)을 모두 결정한다. 세 갈래는 상호 배타.

## 6. 데이터 모델 · 저장

### 6-1. 등록 문서 스키마 (시드 `DY_DOCS_V2`와 호환)

기존 시드 필드 + 등록 전용 필드(`origin`, `formRef`, `linkTarget`, `files`)를 더한다.

```js
{
  id:          "U2-0001",        // 사용자 등록 문서 ID (§6-3)
  origin:      "user",           // user | seed-override  (시드 구분)
  name:        "…",              // 문서명
  menuKey:     "safety",         // 대메뉴(법령 분류)
  daemenu:     "안전보건경영",     // menuKey 라벨(표시용)
  setId:       "A1",             // 세트(없으면 null → 세트 미지정)
  pdca:        "P",              // P|D|C|A
  processType: "전자문서",        // 첨부파일|전자문서|프로그램
  docRole:     "계획",
  legalBasis:  "…", theme:"…", cycle:"연간", applyCond:"상시",
  assignee:    "김도시", dept:"도시과",
  version:     "v1.0", note:"",
  status:      "미시행",          // 미시행|진행|완료 (기본 미시행)
  updated:     "2026-06-23",     // 저장일

  // 처리유형별 (해당 갈래만 채움)
  formRef:     { kind:"standard", formId:"F1" },   // 전자문서
  linkTarget:  "risk",                              // 프로그램
  files:       [{ name:"…", desc:"…", date:"…" }],  // 첨부파일
}
```

### 6-2. localStorage 키 (프로토타입 영속 — `edoc.js`와 동일 전략)

| 키 | 내용 |
|---|---|
| `dy-userdocs-v2` | 사용자 등록 문서 배열 `[{…스키마}]` |
| `dy-docoverride-v2` | 시드 문서 수정 오버레이 `{ "V2-001": {부분필드} }` |
| `dy-doctomb-v2` | 사용자 삭제 표시(tombstone) `["U2-0003", …]` |

### 6-3. ID 채번

- 사용자 문서: `"U2-" + zeroPad(seq, 4)` (예: `U2-0001`). `seq` = 기존 사용자 문서 수 + 1.
- 세트 매핑: 등록 문서의 `setId`는 문서 객체에 직접 보유(시드와 동일). 별도 `DOC_SET` 갱신 불필요 — 조회 시 `setId` 우선 사용.

### 6-4. 병합 규칙 (조회 시) — setlist.js · doc-detail.js 공통

`DYREG.allDocs()`가 단일 진입점이 된다.

```
allDocs():
  base   = DY_DOCS_V2.map(d => applyOverride(d, dy-docoverride-v2))
  base   = base.filter(d => !tombstone.includes(d.id))   // 사용자 삭제 반영
  user   = dy-userdocs-v2.filter(d => !tombstone.includes(d.id))
  return base.concat(user)

setIdOf(doc): return doc.setId ?? DY_DOC_SET_V2[doc.id] ?? null
```

- `setlist.js`/`doc-detail.js`는 `window.DY_DOCS_V2` 직접 참조 대신 `DYREG.allDocs()`,
  `DOC_SET[id]` 참조 대신 `DYREG.setIdOf(doc)`로 치환(최소 수정).

## 7. CRUD 동작 명세 (`window.DYREG` API)

| 함수 | 시그니처 | 동작 |
|---|---|---|
| 등록 모달 | `openCreate(opts?)` | 빈 마법사 모달. `opts.menuKey`로 초기 탭 지정 가능(대메뉴 탭에서 등록 시) |

| 삭제 | `remove(docId)` | user면 배열에서 제거, seed면 `dy-doctomb-v2`에 추가. **확인 모달 후 실행** |
| 조회 | `allDocs()` / `setIdOf(doc)` | §6-4 |
| 단건 | `getDoc(docId)` | 병합본에서 1건 |

- **저장 후**: 호출 화면이 목록이면 `DYSETLIST` 재렌더, 상세면 페이지 리로드.
- **삭제 후**: 상세에서 삭제 시 목록으로 이동, 목록에서 삭제 시 재렌더.

## 12. 수용 기준 (Acceptance Criteria)

1. 업무 목록·이행 목록 툴바의 [+ 업무 등록]으로 마법사가 열린다.
2. STEP1에서 대메뉴 **탭 선택**, PDCA **라디오 선택**, 세트 드롭다운이 대메뉴에 따라 바뀐다.
3. STEP2에서 처리유형 선택에 따라 STEP3 하위 폼이 **세 갈래로 다르게** 표시된다.
4. 전자문서 선택 시 **전자문서 빌더 양식(표준 7종 + 프리셋)** 을 고를 수 있고 미리보기가 보인다.
5. 프로그램 선택 시 **연결 메뉴**를 고를 수 있다.
6. 저장하면 목록의 해당 대메뉴·세트·PDCA 위치에 새 문서가 즉시 나타난다.
7. 새로고침 후에도 등록 문서가 유지된다(localStorage).
8. 등록 문서를 문서 상세/목록에서 **수정·삭제**할 수 있고 결과가 즉시 반영된다.
9. 등록 문서 상세는 처리유형대로 동작한다(첨부=미리보기 / 전자문서=입력폼 / 프로그램=이동).
10. 시드 문서 원본은 삭제·수정해도 데이터 파일이 변하지 않는다(오버레이/tombstone).
