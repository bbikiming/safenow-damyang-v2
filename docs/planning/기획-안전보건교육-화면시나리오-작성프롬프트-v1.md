# 3. 사용자(역할) 정의 — 시나리오의 주인공

권한 등급은 `_공통_권한정의.md` §3 의 6등급이지만, **시나리오는 등급이 아니라 "업무 주체"로 쓴다.**
등급은 각 시나리오 말미에 "권한 근거" 한 줄로만 연결하라.

| 업무 주체 | 대응 등급 | 이 대메뉴에서 하는 일 |
|---|---|---|
| **재난안전과 담당자**(전담부서) | 전담부서(중대재해 총괄) | 집합교육 등록·교육 종료 처리, 채용시교육 일괄 이수처리, 미이수자 독촉, 전 부서 현황 모니터링 |
| **부서 담당자** | 메뉴별 개별 지정(예: 보건소) | 소속 부서 신청(대상자 선택 + 서명파일 업로드), 자체교육 등록·즉시 완료, 기타 교육 등록, 소속 계약직 명단 관리 |
| **관리감독자** | 과장·팀장 | 본인이 교육 대상(12개월 16h)이자, 부서 관리감독자 교육의 운영 대상 |
| **현업 근로자** | (시스템 미사용) | 교육 대상자 — 명단·이수기록의 대상일 뿐 화면을 직접 조작하지 않는다. **이 점을 문서에 명시하라** |
| **열람 계층** | 군수 · 부군수·국장 · 과장·팀장 · 일반 공무원 | 이수현황 읽기전용 열람 |
| **시스템 관리자** | 시스템 관리자 | 메뉴·권한 설정(이 대메뉴 업무 자체는 수행하지 않음) |

주의 — 현재 프로토타입에는 로그인·접근제어가 없어 **전 사용자에게 모든 화면이 동일하게 보인다**(§4 공통 권한 정의).
시나리오는 "실제 운영 시 이렇게 나뉜다"는 목표 상태로 쓰되, 이 괴리를 §6 에 반드시 적어라.

6. 재검토·미결 포인트       — 기획자가 판단해야 할 항목 (§6 참조)

## §3 화면별 구성 작성 규칙
- 화면마다 **"이 화면이 없으면 무슨 업무가 막히는가"**를 한 문장으로 먼저 쓴다.
- 조회 조건(필터)은 나열만 하지 말고 **어떤 업무 질문에 답하기 위한 조건인지** 함께 적는다.
  (예: 이수현황의 '마감 임박 D-30' → "이번 달에 독촉해야 할 대상이 누구인가")
- 액션은 **누가 누를 수 있는지**를 함께 적는다.

## §5 업무 흐름 종합
교육 유형별로 상태가 어떻게 움직이는지, 그리고 **이수시간이 언제 실제로 반영되는지**를 명확히 하라.
- 집합교육: 등록(접수중) → 부서 신청 → 교육 종료 처리 **시점에 신청자 전원 일괄 반영**
- 자체교육 / 기타 교육: 등록·진행 처리 **즉시 반영**
- 채용시교육: 일괄 이수처리 시 반영. 처리 기준일에 따라 정상/지연이 갈림

# 5. 사용자별 시나리오 작성 규칙 (핵심)

각 시나리오는 아래 형식을 지킨다. **정상 흐름만 쓰지 말고 예외·회수 경로를 반드시 짝지어라** —
실제 운영에서 반드시 마주치는 상황이고, 여기서 합의가 빠지면 구축 단계에서 분쟁이 된다.

```
### 시나리오 N. {주체}가 {목적}을 한다
- 전제: (어떤 상태에서 시작하는가)
- 정상 흐름: 화면 → 액션 → 결과 데이터 변화 (번호 목록)
- 예외·회수: 잘못했을 때 어떻게 되돌리는가 / 되돌릴 수 없다면 무엇으로 대체하는가
- 검증 규칙: 시스템이 막는 조건 (필수값·상태 제약)
- 권한 근거: (등급 또는 개별 지정)
```

최소한 아래 시나리오는 모두 포함하라.

**정상 흐름**
1. 재난안전과가 월별 집합 정기교육을 등록하고 부서 신청을 받는다
2. 부서 담당자가 대상자를 선택해 신청하고 서명파일을 올린다
3. 재난안전과가 교육 종료 처리로 신청자 전원의 이수시간을 반영한다
4. 부서가 자체교육을 등록·즉시 완료 처리한다
5. 부서가 기타 교육(5종 분류)을 등록한다
6. 재난안전과가 채용시교육 미이수자를 일괄 이수처리한다
7. 재난안전과가 이수현황에서 미달자를 찾아 독촉한다
8. 부서가 신규 계약직을 명단에 등록한다(직접 등록 / 엑셀 업로드)
9. 관리감독자 정기·기타 교육을 운영한다(현업과 무엇이 다른가)

**예외·회수 경로** — 각각 "무엇이 함께 회수되는가"를 반드시 명시
10. 잘못 등록한 교육을 수정한다 / 삭제한다 (신청·이수기록 연쇄 회수)
11. 부서가 신청을 취소한다 (종료된 교육이면 이수시간도 회수)
12. 잘못 처리한 채용시 이수를 취소한다 (미이수로 복귀)
13. 대상자를 바꿔야 한다 → **수정 화면에서 불가**. 신청 취소 후 재신청(집합) 또는 삭제 후 재등록(자체·기타)으로만 처리
14. 퇴직자가 발생한다 (명단에서 숨김 · 이수 이력은 보존)

# 6. 반드시 §6 에 담을 재검토 포인트

기획자가 판단해야 할 항목이다. 발견한 것을 추가해도 좋다.

1. **화면 정의서 미갱신** — `docs/screen-definitions/` 의 안전보건교육 정의서는 아직 구 "단일 화면 3탭" 기준이다.
   현행 7메뉴 구조로 개정 후 `_build-data.py` 재생성이 필요하다.
2. **메뉴 분리 선결조건** — `docs/archive/안전보건교육-운영고도화-메뉴구조-보완안.md` §5 의 4개 조건
   (담당자·권한 분리 / 명단 원천 / 운영 건수 / 발주처 승인)이 실제로 확인된 적이 있는지.
   현행 7메뉴 구조는 이 확인 없이 소스 스냅샷 이식으로 확정됐다.
3. **메뉴별 권한 미설계** — 현재 edu 7개 메뉴가 동일 프리셋을 공유한다.
   부서 담당자는 "자기 부서만" 신청·등록해야 하는데, 데이터 스코프는 PERM-002 로 Phase 2 이관 상태다.
   → 부서 담당자가 타 부서 신청을 할 수 있는 현 구조가 용인 가능한지 판단 필요.
4. **관리감독자 채용시교육** — 별도 메뉴 없이 '기타 교육' 분류로 흡수돼 있다(결정사항 A-4). 운영상 문제없는지.
5. **부서이동 승계** — 결정사항 B-2 미확정. 사이클·이수시간이 부서 이동 시 어떻게 따라가는지.
6. **프로토타입 ↔ 목표 상태 괴리** — 로그인·접근제어 미구현으로 현재는 전원이 모든 화면·액션 사용 가능.
7. **기준일 고정** — 시연 데이터가 `DYEDU.TODAY = '2026-07-16'` 기준으로 계산된다. 실제 운영 시 실시간 날짜와의 차이.

# 7. 작성 원칙 (반드시 지킬 것)

- **기획 순수성** — 이 문서는 기획 산출물이다. 함수명·변수명·파일 경로·CSS 클래스 같은
  **구현 세부는 쓰지 마라.** 화면은 화면명으로, 데이터는 업무 용어로 쓴다.
  (예외: §1 근거 파일 목록과 §6 재검토 포인트에서 파일을 특정할 때만 경로 허용)
- **추측 금지** — 코드나 SoT 문서에서 확인되지 않은 동작은 쓰지 않는다.
  확인이 안 되면 `[확인필요]` 로 표시하고 §6 에 올린다.
- **법적 근거 병기** — 필요시간·주기는 근거(산안법 조항 또는 담양군 운영 방침)를 함께 적는다.
  결정사항 문서에 근거가 명시된 항목만 인용하고, 없으면 "담양군 운영 방침(결정사항 v1 A-N)"으로 표기한다.
- **표를 적극 활용** — 화면 8개 대비, 사용자 6종 × 화면 접근 매트릭스는 표가 훨씬 읽힌다.
- 분량은 제한하지 않되, **§4 시나리오가 문서의 절반 이상**을 차지해야 한다.
