## 배경

이 저장소(담양군 중대재해 통합관리 시스템 프로토타입 v2 — 빌드 없는 정적 HTML + 전역 IIFE JS + css/style.css·v2.css + Tailwind CDN)에 대해 2026-07-16에 디자인 시스템 전수 점검을 완료했고, 결과와 조치 플랜이 **`../planning/기획-디자인시스템-점검-v1.md`에 정리되어 있다. 이 문서가 본 작업의 단일 기준(SoT)이다.** 시작 전에 반드시 전체를 읽어라. `CLAUDE.md`의 UI 규칙(단일 모달·업로드 드롭존·조직도 단일 출처)도 함께 준수한다.

이슈 27건(심각 3·중간 15·경미 9)이 HC-xx(하드코딩)·CP-xx(컴포넌트 일관성)·RS-xx(반응형) ID로 정리돼 있고, 조치는 6단계다:

- **Phase 0 — 표준 정의**: `--surface`/`--surface-alt`/그림자·오버레이 토큰 신설, z-index 레이어 토큰 6단계, 브레이크포인트 4단계(1280/1024/768/560) 통일안, 상태 라벨→tone 매핑표, CLAUDE.md 규칙 증보
- **Phase 1 — 토큰 일원화**: v2.css 원시 hex 120회·font-size 리터럴 ~260회·radius 72회 → 토큰 치환, tailwind-config `primary.50` 오기 정정(#DFF1E9→#F1FAF5), JS/HTML 원시색·z-index 리터럴 치환
- **Phase 2 — 심각 위반 해소**: evl-eval·evl-settings 자작 모달 → `DYV2.openModal` + 적층 픽커 모달 폐지(`.org-inline`/ORGPICK 인라인 전환), menu.js body-append 오버레이 4곳 정비 + `closeModal()` 정리 목록 보완
- **Phase 3 — 컴포넌트 통합**: 상태 배지 5계열→1, select 2계열→1(별칭 후 점진 치환), 테이블·빈상태·토스트·드롭존(menu.js `opn-dropzone`) 통합
- **Phase 4 — 반응형 보강**: 브레이크포인트 재배열, `.card-body` overflow 사각지대(768~1023px) 해소, 모달 내부 min-width 표 잘림, rsk 계열 @media 신설, 터치 타깃·11px 폰트 상향
- **Phase 5 — 통합 검수**: 375/768/1024px 3폭 전 화면 점검 + grep 정적 검사(원시 hex·px 폰트·리터럴 z-index 잔존 0) + `기획-디자인시스템-점검-검수-v1.md` 작성

의존 관계: 0 → (1‖2 병행 가능) → 3 → 4 → 5.

## 진행 방식

1. **착수 전 현행화 검증 (중요)**: 점검 이후 위험성평가 재설계 v1.1(rsk-proc/exec 메뉴 제외)과 안전보건교육 재설계(edu.html→edu-* 7화면, css/edu.css 신설)가 있었다. 점검 문서가 지목한 파일:라인이 현행 코드와 다를 수 있으니, **각 이슈 착수 전 grep으로 위치를 재확인**하고, 폐기된 화면(구 rsk-proc/exec 등)에만 해당하는 이슈는 "해당 없음(폐기됨)"으로 기록 후 건너뛴다. 신설된 `css/edu.css`·edu-* 화면은 점검 범위 밖이었으므로 같은 기준(토큰·브레이크포인트·표준 클래스)으로 **추가 점검해 동일 플랜에 편입**한다.
2. Phase 순서대로 진행하고, **Phase당 1커밋**으로 완결한다(커밋 메시지에 해소한 이슈 ID 명시). 사용자가 특정 Phase만 지시하면 그 범위만.
3. Phase 0의 산출물(토큰 정의·매핑표·CLAUDE.md 증보)을 먼저 보여주고 확정한 뒤 Phase 1 치환에 들어간다.
4. 각 Phase 완료 시 점검 문서의 해당 검수 기준으로 자체 검증한다. Phase 1 치환 후에는 대표 화면(index·rsk-list·evl-eval·health-exam·menu)을 브라우저 프리뷰로 육안 대조한다.

## 불변 제약

- `DYV2.ORG`의 **uid 변경 금지**(권한 시드가 참조).
- 모달은 `DYV2.openModal/closeModal`만 사용, 적층 모달 금지, 조직도는 `.org-inline` 인라인 패널(CLAUDE.md §1).
- 토큰 치환은 "픽셀 동일"이 아니라 "토큰 정합"이 목표 — 근사색이 토큰값으로 수렴하며 색이 미세하게 변하는 것은 의도된 결과다.
- `onepager-docflow.html`·`screen-definitions.html`은 독립 문서 성격이므로 치환 대상 제외.
- evl-eval 모달 재편(2-1)은 최대 리스크 — 평가 매트릭스↔평가자 선택 플로우(자동 선정 안내·임시저장 분리)를 반드시 회귀 검증.
- 사용처 많은 클래스(select·배지)는 즉시 삭제 대신 별칭 유지 후 점진 치환.

우선 점검 문서와 CLAUDE.md를 읽고, 1번(현행화 검증) 결과와 Phase 0 표준안을 제시하는 것부터 시작해 줘.
