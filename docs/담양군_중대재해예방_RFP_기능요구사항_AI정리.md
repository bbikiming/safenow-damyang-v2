## 0. 사용 방법
이 문서는 RFP의 기능 요구사항(SFR)을 AI가 읽기 쉽도록 구조화한 Markdown이다. 원문 요구사항을 기준으로 `id`, `name`, `definition`, `functional_requirements`, `deliverables`, `integrations`, `tags`, `source_page`를 분리했다. 설계나 제안서 작성 시에는 각 SFR의 원문 요구사항을 우선 기준으로 삼고, 하단의 `AI 해석용 구조화 메모`는 기능 분류와 화면 도출을 위한 보조 정보로 사용한다.

## 4. 공통 기능 패턴

| 패턴 | 설명 |
|---|---|
| CRUD 반복 | 다수 기능에서 등록, 조회, 수정, 삭제가 반복된다. 공통 컴포넌트와 목록/상세/수정 패턴을 표준화해야 한다. |
| 점검표 생성 | 위험성평가, 의견청취, 의무이행, 도급/용역/위탁, 인력·장비·예산 등에서 점검표 생성과 결과 등록이 반복된다. |
| 결과 상태값 | 완료/진행/미시행 상태 구분이 SFR-007, SFR-010 등에서 명시된다. 상태 코드를 공통화하는 것이 적합하다. |
| 증빙자료 | SFR-005, SFR-008, SFR-013, SFR-014 등에서 첨부파일 또는 증빙서류 관리가 필요하다. |
| 알림 | SFR-005, SFR-017, SFR-020에서 담당자 알림, 문자, Push, 알림 내역 조회가 연결된다. |
| 권한 | SFR-001과 SFR-015에서 메뉴/기관/담당자별 권한이 명시된다. |
| 통계 | SFR-007, SFR-018, SFR-019, SFR-020에서 통계 집계·시각화·다운로드가 요구된다. |

## 5. 상세 기능 요구사항


### SFR-001 - 프로그램 개발 공통 사항

```yaml
id: SFR-001
type: 기능 요구사항
name: 프로그램 개발 공통 사항
definition: 공통구현 요구사항
source_page: 1
tags:
  - 공통
  - 표준
  - UI/UX
  - 권한
  - 확장성
  - 웹접근성
  - 웹표준
integrations:
  - open API 연동 가능성 고려
deliverables:
  - 디자인 가이드 제시(메인 디자인 제안 시 제시)
```

#### 기능 요구사항
- 담양군의 기능 요구사항과 업무 프로세스를 적용하여 개발한다.
- 전자정부표준프레임워크 최신버전을 적용하여 유지보수 효율성과 안정성을 확보한다.
- 제안내용은 신의와 성실을 바탕으로 작성하고, 필요 시 제안서 내 정보를 입증할 수 있는 자료를 제공한다.
- 담양군 중대재해예방 통합관리시스템의 디자인 가이드를 제시한다. 대상 영역은 메인, 서브, 메뉴, 데이터 영역 등을 포함한다.
- 글로벌 웹 표준 HTML5, CSS3를 반영한다.
- OSMU(One Source Multi Use) 구현의 관리 효율성을 확보한다.
- 다양한 웹브라우저를 지원한다.
- 데이터 분석을 효율적으로 수행할 수 있도록 사용자 중심의 편의기능과 화면을 제공한다.
- 각 기능 및 관리메뉴별 권한 부여가 가능한 관리자 설정 기능을 제공한다.
- 사용자는 권한을 부여받지 않은 메뉴에 접근할 수 없어야 한다.
- 시스템 목적에 맞게 구축하고, 상호연계성, 향후 확장성, 유연성 확보방안을 제시하여 설계에 반영한다.
- 사업수행 시 제안된 개발방법론에 의해 사업을 수행한다.
- 전자정부 웹사이트 UI/UX 가이드라인을 준수한다.
- 콘텐츠 접근성 향상을 위한 효율적인 내비게이션을 구성한다.
- 개발 소프트웨어는 공통 컴포넌트 수준의 개발 표준을 준수한다.
- 시스템 교체 등 다양한 디바이스 환경에서 일관성 있게 서비스할 수 있도록 구축한다.
- ActiveX 및 비표준 Flash 파일을 배제한다.
- 웹 접근성, 웹 표준, 개발보안 등 관련 법령, 지침, 표준을 준수한다.
- 시스템 운영 시 기능별 활용 여부를 점검할 수 있는 기능을 제공한다.

```yaml

  - 관리대상
  - 시설물
  - FMS
```

#### 기능 요구사항
- 중대재해처벌법 시행령에 근거하여 중대재해처벌법 적용대상에 대한 담양군 전용 관리시스템 프로세스를 정립하고 기능을 개발한다.
- FMS 내 시설물 현황 및 시설물안전법에 따른 1, 2, 3종 시설물 분류를 자동 연계한다.
- 연계된 시설물 정보를 구축시스템에서 관리하고 점검할 수 있도록 개발한다.
- 관리대상 시설물 정보가 변동되는 경우, 예를 들어 FMS 내 시설물 신규 등록 등이 발생하면 연계 기능을 통해 자동 현행화한다.

## 6. AI 해석용 구조화 메모

### 6.1 화면 도출 시 우선 고려할 공통 화면
- 목록/검색 화면: 기관, 부서, 시설, 담당자, 기간, 상태, 분류 조건을 공통 검색 조건으로 설계
- 상세/수정 화면: 원문에서 반복되는 등록, 조회, 수정, 삭제 기능을 공통 CRUD 패턴으로 설계
- 점검표 생성/작성 화면: 점검표 자동 생성, 점검 결과 입력, 상태값, 조치결과, 증빙첨부 포함
- 첨부파일/증빙자료 관리 영역: 파일 다중 첨부, 설명, 삭제, 조회 기능 포함
- 결재/승인 화면: 온나라 또는 전자결재 연동 대상과 내부 승인 대상 구분
- 알림 설정 화면: 담당자 선택, 발송 채널, 즉시/예약, 전체/개인/기관 단위 발송 구분
- 통계/대시보드 화면: 기간 조회, 상태별 집계, 접수/조치율, 엑셀 다운로드 포함

### 6.2 주요 데이터 엔티티 후보
- 기관/부서/담당자/권한
- 관리대상 시설물/시설물 분류/시설물 점검등급
- 안전계획/의무이행 점검/보고서
- 안전보건 목표/경영방침/관련근거/법령/고려사항
- 전담조직/조직도/업무/예방활동 대상
- 유해·위험요인/위험성평가/감소대책/개선실적/점검결과
- 인력/시설/장비/예산편성/예산집행/집행률/교육이수/증빙자료
- 안전관리자/보건관리자/산업보건의/공사정보/선임현황
- 종사자 의견/제안/오프라인 접수/조치계획/조치경과/조치결과
- 도급·용역·위탁 사업/안전보건 확보 절차/서약서/증빙서류
- 알림/SMS/Push/메일/발송이력/예약발송
- 통계/대시보드 카드/차트/달력 일정/알림 메시지 내역
- 제증명/보고서/워터마크/위조방지

### 6.3 구현 우선순위 판단용 의존관계
- 권한/인증(SFR-001, SFR-015)은 모든 관리 기능의 선행 기반이다.
- 조직도/담당자 연동(SFR-006, SFR-015)은 점검자 지정, 알림, 결재, 권한 설정의 기반이다.
- 시설물 관리대상 정보(SFR-002, SFR-016)는 안전계획, 점검, 대시보드의 핵심 기준 데이터다.
- 점검표 엔진은 SFR-004, SFR-007, SFR-008, SFR-011, SFR-012, SFR-013, SFR-014에서 반복 사용된다.
- 알림체계(SFR-017)는 목표/방침 등록(SFR-005), 신규 정보 등록, 대시보드 알림 메시지(SFR-020)와 연결된다.
- 통계(SFR-018)는 위험성평가 상태, 유해위험요인 신고 조치율, 시기도래 업무, 대시보드 시각화와 연결된다.

### 6.4 불명확하거나 협의가 필요한 항목
- SFR-009 평가기준의 세부 항목, 배점, 평가주기, 평가대상 범위는 담당자 협의가 필요하다.
- SFR-018 현황 및 통계관리 기능은 담당자와 협의하여 진행한다고 명시되어 있어 통계 항목 확정이 필요하다.
- SFR-020 대시보드의 실제 지표명, 데이터 산식, 차트 유형, 카드 구성은 디자인 시안 및 담당자 협의가 필요하다.
- SFR-017 Push 알림의 URL response 방식 세부 기술규격, 수신 대상, 발송 로그 정책 확정이 필요하다.
- 전자결재 연동은 온나라 전자결재와 일반 전자결재 표현이 혼재되어 있어 연동 대상과 승인 프로세스 구분이 필요하다.
- 차세대 e호조 외 타 회계 시스템을 사용하는 경우 직접 등록 범위와 데이터 항목 정의가 필요하다.
