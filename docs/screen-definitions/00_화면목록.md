## 1. 분해·코드 부여 기준 (Decomposition & Code System)

### 1-1. 화면 분해 기준 (decomposition)
중복 분석 결과를 통합한 뒤, 아래 규칙으로 "독립 화면" 단위를 확정했다.

1. **URL이 다르면 별도 화면** — 파일/라우트가 다르면 별도 화면으로 분리한다.
2. **목록과 상세는 별도 화면** — 같은 도메인이라도 목록(list)과 상세(detail)는 분리한다. (예: 위험성평가 목록 ↔ 공정별 진행 상세 ↔ 위험성 추정 실시)
3. **`menu.html?m=` 의 각 대메뉴는 별도 화면** — 동일 셸(menu.html)이라도 쿼리 `m=policy|org|opinion|contract|comply` 각각을 독립 업무화면으로 분리한다. (`m=risk|hazard|improve|edu`는 즉시 전용 화면으로 리다이렉트되므로 menu.html에서 그리지 않고, 각 전용 화면 SCR로 집계. **개선조치(`m=improve`)는 2026-07-13 재편으로 위험성평가 그룹 정본 화면 `rsk-imp.html`로 이동**)
4. **쿼리/탭/상태 변형은 화면 성격이 독립적일 때만 분리** — 한 파일이 쿼리·상태·`data-*` 토글로 사실상 다른 업무화면을 그리는 경우 분리한다.
   - `base-targets.html?tab=biz|facility|material` → 3개 화면(SCR-BASE-001~003)
   - **위험성평가 재편(2026-07-13):** 구 위험성평가 화면군(목록·정기 회차 상세·4M 상세·체크리스트·유해위험요인별 O·X·등록폼·생성 마법사·투어 · 유해위험요인 관리 목록·상세)을 레퍼런스 정합 흐름으로 통합해 **작업공정 관리(SCR-PRC-001) → 위험성평가 목록(SCR-RISK-001) → 공정별 진행 상세(SCR-RISK-002) → 위험성 추정 실시(SCR-RISK-003) → 개선조치 목록·상세(SCR-IMP-001·002)** 로 재편했다. 평가방법 3종(빈도·강도 / 3단계 판단법 / 체크리스트법)은 위험성 추정 실시 1개 화면의 기법 전환으로 흡수한다.
   - **안전보건교육 재이식(2026-07-20~21):** 구 `edu.html` 단일 3탭(일정/이수/과정)은 폐지되고, **독립 GNB 그룹 `edu` · SNB 3뎁스 7메뉴 + 상세 1**로 재이식되었다. 이에 따라 구 SCR-EDU-001~003(edu.html 탭)을 폐기하고 현행 6화면 **정기교육(SCR-EDU-001, `edu-reg.html`·관리감독자 `edu-sup.html`) · 채용시교육(002) · 기타 교육(003, `edu-etc.html`·관리감독자 `edu-sup-etc.html`) · 정기교육 상세(004, `edu-reg-detail.html`) · 이수현황(005, `edu-status.html`) · 근로자 명단 관리(006)** 로 재정의했다. 관리감독자 정기·기타(`edu-sup*.html`)는 동일 화면의 대상 모집단·사이클 변형이라 각 정기·기타 정의서에 흡수 기술한다. `edu.html`은 `edu-status.html` 리다이렉트 스텁이라 별도 화면으로 두지 않는다.
   - **위험성평가 정기·수시 분리(2026-07-21):** 재설계 v1.1로 위험성평가 그룹이 **정기(`rsk-list.html`)·수시(`rsk-occ.html`)·개선조치**로 확정되었다. 정기는 목록·상세·보고서 검수·조치기한 전달을 한 화면에 통합하고(SCR-RISK-001), 수시는 신규 화면으로 분리(SCR-RISK-004)했다. 이에 따라 **위험성평가 상세(`rsk-detail.html`, SCR-RISK-002)는 하위호환 리다이렉트 스텁**, **위험성 추정(`rsk-exec.html`, SCR-RISK-003)·작업공정 관리(`rsk-proc.html`, SCR-PRC-001)는 폐지(파일 보존·메뉴 제외)** 로 표기한다.
   - `doc-detail.html`은 `processType`(첨부파일/전자문서/프로그램)에 따라 좌측 본문이 3분기되고 입력 유무가 달라 **3개 화면(SCR-DOC-004~006)으로 분리**. v1/v2(`?v=2`)는 데이터 소스/관계 산출만 바뀌므로 분리하지 않고 각 정의서 내 분기로 기술.
   - `docs-preset.html`(테이블형) ↔ `docs-exec.html`(격자 카드형)은 동일 `setlist.js` 컴포넌트지만 URL·표현형이 달라 **별도 화면(SCR-DOC-002, SCR-DOC-003)**.
5. **독립 입력·검증·저장 팝업은 별도 팝업 정의서** — e-Doc 표준 폼 F1~F7은 자체 입력·상태머신·저장(localStorage)을 가지므로 **개별 팝업 정의서(SCR-EDOC-001~007)**로 분리한다. 온나라 결재 요청 팝업·알림 발송 시뮬·기준문서 업로드 모달도 독립 동작이라 별도 정의(SCR-EDOC-008~010).
6. **단순 확인/안내 팝업은 부모 이벤트로 흡수** — 등록 완료 모달, 설정 가이드 모달, 결재 이력 모달, 적용대상 선택 모달, 미리보기 모달 등 입력·검증·저장이 없는 표시용 팝업은 별도 화면으로 만들지 않고 **부모 화면 정의서의 버튼/이벤트 항목으로 흡수**한다.
7. **전역 공통 UI는 SCR-COM으로 집약** — 헤더/GNB/LNB/알림 드롭다운/모달 인프라/토스트/페이지 크롬은 모든 페이지에 자동 주입되므로 전역 공통 화면(SCR-COM-001~007)으로 1회만 정의한다.

### 1-2. 화면ID·코드 체계 (codeSystem)
- **SCR ID를 정의서 정본(正本) 식별자**로 사용한다. 영역별 접두어 + 3자리 일련번호(001부터).
- 기존 **SFR 코드는 폐기하지 않고 매핑 컬럼("기존 SFR")으로 병기**한다. SFR은 `layout.js` NAV 레지스트리의 `screen` 필드와 `common.js` MENUS의 `sfr` 필드에서 추출했으며, SCR↔SFR은 다대일·일대다가 공존한다(예: SCR-SBM-002 ↔ SFR-006·009·010). SFR이 없는 화면(업무문서 일부·정보센터·프리셋 관리 등)은 "없음"으로 표기한다.

| 접두어 | 영역 |
|---|---|
| SCR-DASH | 대시보드 |
| SCR-BASE | 기본정보(관리대상·일괄등록) |
| SCR-SBM | 안전보건관리체계 대메뉴(menu.html?m=) |
| SCR-PRC | 작업공정 관리(위험성평가 그룹) |
| SCR-RISK | 위험성평가(정기·수시·상세) |
| SCR-IMP | 개선조치(위험성평가 그룹·정본) |
| SCR-EDU | 안전보건교육 |

> **접두어 개정(2026-07-13):** GNB를 **위험성평가 그룹**(작업공정 관리·위험성평가·개선조치)으로 재편함에 따라 접두어를 정비했다. 구 `SCR-HAZ`(유해·위험요인 관리)는 **작업공정 관리(SCR-PRC)로 통합**되어 폐지하고, 개선조치는 안전보건관리체계 대메뉴(구 SCR-SBM-005)에서 위험성평가 그룹 정본(SCR-IMP)으로 이관했다.
| SCR-DOC | 업무문서(기준문서함·세트목록·문서상세) |
| SCR-RPT | 통계·보고 |

### 1-3. 팝업 흡수/승격 결정 (popupPromotion)
- **승격(별도 정의서):** e-Doc F1~F7 폼(독립 입력·검증·상태머신·localStorage 저장), 온나라 결재 요청 팝업, 알림 발송 시뮬, 기준문서 업로드 모달, 공통 모달 인프라(`#v2-modal`), 전역 알림 드롭다운, 전역 토스트.
- **흡수(부모 이벤트):** 등록 완료/설정 가이드/근거관리/점검표 편집/적용대상 선택/조직도 선택/알림대상 설정/결재 이력 모달, SIF 추천 모달(상세 화면 내 동작), 위험요인 추가 모달, 개선조치 입력 모달, 양식 미리보기 모달, 브라우저 alert/confirm 등 — 모두 해당 부모 화면 정의서의 "버튼·팝업" 절에 기술.

## 2. 전역 공통 사실 (전 화면 적용)
- **권한:** `layout.js`/`common.js` 및 전 HTML에 로그인·세션·역할(role)·접근제어 분기 로직이 **전무**. 사용자 필 '박안전 님 / 안전건설과', 작성자 '박안전', 결재선 '팀장→과장→부군수'는 모두 하드코딩. 모든 메뉴·버튼이 누구에게나 동일 노출. ('시스템 관리 = 관리자 전용'은 주석뿐, 강제 없음)
- **유효성:** 공통 인프라 및 대부분 폼에 필수값·형식·범위 검증과 오류 메시지가 거의 없음. e-Doc `collect()`는 무검증 수집. 일부는 미입력 시 기본값 대체(제목→'제목 없는 의견', 파일설명→'(설명 없음)')로 차단이 아님.
- **목업/시뮬레이션:** 업로드·다운로드·동기화·재시도·페이지네이션·검색/필터·'모두 읽음'·온나라 연계·문자/메일 발송 등 다수 액션이 `DYV2.toast()` 또는 `DYLayout._soon()` 토스트(또는 alert)만 수행. 데이터는 `data.js`/`sets-data*.js` 하드코딩 또는 localStorage 영속(서버 없음).
- **데이터 정합성 주의:** `data.js` 문서 424건에 `status` 필드가 없어 `stats.html` 이행 현황표·`complianceRate()`가 전부 0%로 출력됨(SCR-RPT-001 확인필요 사항).

## 4. 집계
- **총 화면 수:** 머신 인벤토리(`_inventory_rows.json`) 기준 **57개**. (시설물 안전관리(FAC) 화면군은 머신 인벤토리에만 집계되어 위 §3 표는 큐레이트 뷰다.)
  - **위험성평가 재편(2026-07-13):** 구 위험성평가 8화면 + 유해·위험요인 관리 2화면 + 안전보건관리체계 개선조치 1화면(총 11)을 **위험성평가 그룹 6화면**(작업공정 관리 1 · 위험성평가 목록·상세·추정 3 · 개선조치 목록·상세 2)으로 통합. 유해·위험요인 관리(SCR-HAZ)는 작업공정 관리(SCR-PRC)로 흡수 폐지, 개선조치는 안전보건관리체계(구 SCR-SBM-005)에서 위험성평가 그룹 정본(SCR-IMP)으로 이관.
  - v2 갱신(2026-07-08): 시스템관리에 메뉴 관리(SCR-ADMIN-005)·권한 관리(SCR-ADMIN-006) 신설.
- **팝업 정의서:** 10개(SCR-EDOC 7종 폼 + 온나라·notify·업로드 모달) + 전역 팝업류(SCR-COM-004/005/006) — e-Doc 7폼은 독립 입력·검증·저장으로 승격, 단순 확인 팝업은 부모 흡수.
- **확인필요(undefinedPolicies) 총합:** 위 표 "확인필요 개수" 합계 — 권한(전 화면 미구현)·유효성(검증 부재)·목업 동작(저장/연계/검색/페이지네이션)·상태(로딩/에러/읽기전용 부재)·데이터 정합성(status 필드 부재로 통계 0%) 등이 핵심 미정 항목.

> 참고: `onepager-docflow.html`은 문서 흐름 설명용 정적 원페이저로 업무 기능 화면이 아니어서 인벤토리에서 제외함.
