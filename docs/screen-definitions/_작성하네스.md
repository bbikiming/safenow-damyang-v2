# 화면 정의서 작성 하네스 (v1.1)

> 목적: 모든 화면 정의서가 **SCR-DOC-007(개발 확인 완료) 수준의 "기획 관점" 품질**로 *일관되게* 나오도록 하는 작성 지침·검증가능 체크리스트.
> 역할 분담: **형식 규칙(구조·코드금지)은 `_규칙.md` / `_lint.py`가 자동 강제**(저장 시 훅 검사). 본 문서는 그 위에 **깊이·정확성·일관성(=기획 품질)** 을 보장한다. v1.1은 "해야 할 것"을 *예/아니오로 판정 가능*하게 못박았다(린트는 "하지 말 것"만 막는다).

---

## 0. 골드 스탠다드

- **기준 문서: `SCR-DOC-007_업무등록마법사.md`** — 개발팀이 "이 품질이면 바로 착수 가능"으로 확인한 문서. 작성 전 반드시 한 번 펴 본다.
- 골드를 골드로 만든 4요소(= 본 하네스가 강제하는 것): ① §1 4요소 오프너 ② §3 **분기 로직·엣지케이스** ③ §4 영역별 **상태(빈/로딩/오류 + 업무 단계상태)** ④ §7 **화면-고유** 미결.

---

## 1. 절대 원칙 3가지

1. **기획 관점만** — "이 화면이 *무엇을 제공하는가*(의도) + *화면 설계*"를 업무 용어로. 함수·DOM·저장방식 등 **구현은 개발이 정한다**(쓰지 않는다).
2. **프로토타입이 진실의 출처** — 추측 금지. 실제 화면을 열고/코드를 읽고 적는다. 문서와 프로토타입이 어긋나면 **문서가 틀린 것**(화면이 갱신됐으면 반드시 현행화).
3. **미확정은 숨기지 않는다** — 본문에 흩지 말고 `§7`에 `담당·기한`과 함께. 본문에 `[확인필요]` 금지.

---

## 2. 작성 절차 (관찰 산출물 → 1:1 대응)

1. **관찰(산출물 필수)** — 대상 화면(URL)을 실제로 열고 관련 코드를 읽어, 먼저 **관찰 목록**을 만든다: `진입점 / 영역 / 버튼·동작 / 메시지 / 상태(빈·로딩·오류·단계) / 입력 폼 / 다른 화면 연계`. 읽은 URL·파일을 기록한다.
2. **골드 정렬** — `SCR-DOC-007` + 템플릿(`_화면정의서_템플릿_v6.md`) 펴고 7섹션 골격 세움.
3. **1:1 대응 작성** — 관찰 목록의 모든 버튼·동작이 §4에 빠짐없이 들어갔는지 1:1로 대응시킨다(누락 0). 프로토타입에 없는 것은 쓰지 않는다.
4. **분기·미결 도출** — §3에 분기·엣지케이스(§3 기준), 확정 안 된 모든 것은 §7에 화면-고유로.
5. **자가검증 + 린트** — 아래 **§5 체크리스트** 전 항목 "예" → `python3 docs/screen-definitions/_lint.py <파일>` 위반 0 → `python3 docs/screen-definitions/_build-data.py` 재생성.

## 3. 섹션별 깊이 기준 (골드 수준 · 판정 가능)

- **§1 화면 정보** — 표 5필드 + 목적 문단. 목적 문단은 **4요소를 반드시 포함**: ① 무엇을 제공 ② 왜(법적·업무 목적) ③ 구성 영역 몇 개 ④ 핵심 연계(어느 화면과). 추상어 금지("관리한다"(X) → "등록·점검 후 부적정 항목을 개선조치로 자동 연계한다"(O)).
- **§2 권한 정보** — 4역할(시스템관리자/담당자/부서장/열람자) 표 + "공통 권한 정의서 §6 `[SCR-ID]` 행 참조" 명시. **표기 규약**: 확정 전 셀은 `(제안)` 접미사. **역할 차등이 없으면** "현재 역할별 노출 차이가 없다"를 한 줄 명시하고 *무엇으로 갈리는지*(예: 문서 출처·상태)를 적는다. 권한 확정은 §7로.
- **§3 진입/이탈 + 시나리오 + 분기·엣지** — 진입·이탈을 **빠짐없이**(메뉴·알림·딥링크·타 화면 링크·작업 후 복귀). 시나리오는 *사용자 행동 → 화면 반응* 단계별. **추가 필수**: 선택·입력에 따라 후속 단계/항목/동작이 달라지면 **의존관계를 명시**(골드의 "처리유형 선택 → 상세 폼 분기"). 취소·재선택·상태 전환 시 **기존 입력의 유지/초기화**를 명시(골드의 "처리유형 변경 시 직전 입력 비움"). 빈/0건/오류 경로도.
- **§4 화면 구성별** — 화면을 **영역으로 쪼개** 각 영역마다 ① `구성` ② `트리거 / 동작 결과 / 이동·메시지` 표 ③ `상태`. **"상태"는 두 가지를 모두**: (a) 빈/로딩/오류(해당 시) (b) **업무 단계상태**(예: 미시행→진행→완료, 작성중→등록완료→확정). "정상 표시" 한 줄은 **불합격**. 프로토타입의 버튼·동작은 **하나도 빠뜨리지 않는다**.
- **§5 유효성 검사** — 입력 폼 있으면 `입력 항목 / 필수 / 형식·길이 / 오류 안내` 표. **조건부 필수**가 있으면 조건과 함께(골드: "문서명은 처리유형별 조건부 필수"). 입력 폼 없으면 **"해당 없음" + 사유**(예: "조회 목록이라 저장형 입력 없음")와, 입력 검증이 일어나는 연계 화면 SCR-ID를 가리킨다. 현재 검증 로직이 없으면 사실대로 + 제안은 `(제안)`.
- **§6 정책** — 업무 규칙 + 확정 필요 기준. 각 항목은 *현상 → 무엇을 정해야 하는가(§7 연결)*. **관련 화면은 SCR-ID로 상호참조**.
- **§7 미결사항** — `ID | 항목 | 확인 내용 | 담당 | 기한`. **화면을 실제로 뜯어야 나오는 화면-고유 항목**을 빠짐없이(범용 "저장 방식 확정"만으로 채우지 말 것). 보통 6개 이상이며, 단순 조회 화면이라도 권한·검색/정렬·빈상태·다운로드 등 **최소 5개**. 모든 행에 `담당·기한`(예: "개발 착수 전 / 차기 기획 회의 / 실서비스 단계").

## 4. 정형 요소 (문서마다 동일 — 못박음)

- **헤더 메타라인** — 제목 다음 줄에 항상:
  ```
  # 화면 정의서 — <화면명>

  > 화면 ID: <SCR-ID> · 화면 유형: <유형> · 기획 관점(v6)
  > 확정이 필요한 항목은 §7 미결사항에 모았습니다.
  ```
- **§7 ID 접두어 규칙** — 화면 도메인 **대문자 약어(2자 이상) + 하이픈 + 번호**. **접두어에 숫자를 넣지 않는다.** 예: `REG-01`(등록), `DOC-01`(업무목록), `RISK-01`, `SBM-01`, `CHK-01`(점검표). ✗ `EDOC005-01`(접두어에 숫자 → 자동검사에서 누락됨), ✗ `F5-01`(대문자 1자).
- **상호참조** — 본문에서 다른 화면을 가리킬 땐 이름과 함께 SCR-ID(예: "업무 등록은 SCR-DOC-007", "문서 상세는 SCR-DOC-004~006").

## 5. 자가검증 체크리스트 (제출 전 전부 "예" — 정성표현 금지, 예/아니오로)

- [ ] 헤더 메타라인 2줄(§4 형식)이 있다
- [ ] §1 목적 문단에 4요소(무엇·왜·영역수·핵심연계)가 **모두** 있다
- [ ] §2에 `(제안)` 표기가 쓰였고, 역할 차등이 없으면 그 사실 + 무엇으로 갈리는지 한 줄이 있다
- [ ] §3 진입·이탈이 **실제 연결 화면과 일치**한다(프로토타입 확인)
- [ ] §3에 **분기 로직**(선택→후속 변화) 또는 "분기 없음"이 명시돼 있다
- [ ] §3에 **엣지케이스**(취소·재선택·상태전환 시 입력 유지/초기화, 빈/0건/오류)가 있다
- [ ] §4 **모든 영역**에 구성 + 트리거 표 + 상태가 있다
- [ ] §4 각 영역 "상태"에 빈/로딩/오류(해당 시) **그리고** 업무 단계상태가 있다(한 줄 "정상 표시" 아님)
- [ ] 관찰 목록의 **버튼·동작이 §4에서 누락 0**, 프로토타입에 없는 기능을 지어내지 않았다
- [ ] §5가 입력 폼 유무에 맞다(없으면 사유 + 연계 SCR-ID, 있으면 조건부 필수 반영)
- [ ] §6 각 정책이 현상→확정기준 형태이고 관련 SCR-ID 상호참조가 있다
- [ ] §7 항목이 **화면-고유**이며 5개 이상, 모든 행에 담당·기한이 있다
- [ ] §7 ID 접두어가 §4 규칙(대문자 약어+숫자 없는 접두어)을 따른다
- [ ] `[확인필요]`가 본문에 없다(전부 §7)
- [ ] **코드 토큰이 없다**(§6) → `_lint.py` 통과

## 6. 코드 토큰 금지 (린트가 차단 — 미리 피한다)

기획 문서에 쓰지 않는다(쓰면 자동 검사가 막는다):

- 함수·메서드명(예: `renderBody()`, `EDOC.openForm(`), 내부 데이터·전역·프리셋 키명(예: `DY_DOCS`, `CHECKLIST_PRESETS`, `default`/`scorelist` 같은 키)
- DOM 요소 ID(`#xxx-yyy`), CSS 클래스(`sl-...`, `chip-mini`), `data-*`, `placeholder` 등 속성/UI코드어
- 저장 API·키(`localStorage`, `dy-edoc-v1`)
- 코드 연산자·체이닝·수식(`===`, `||`, `&&`, `.filter(`, `.map(`, `Math.`)
- 소스 파일:라인 참조(`menu.js L12`)
- 보안·코드품질어(`XSS`, `데드코드`, `stale`, `try-catch`, `이스케이프`, `readonly`)

→ **업무 용어로 바꾼다**: "읽기전용 표시", "목록에서 거른다", "저장한다", "한 명만 선택", "입력 도움말", "기본 점검항목 세트" 등.
→ 허용: URL 경로(`docs-preset.html`, `menu.html?m=policy`), 업무 개념(PDCA·세트·처리유형·작성중/등록완료/확정).

## 8. 지시 방법 (instructable)

- **신규 작성**: `/screen-def <SCR-ID 또는 화면/URL 설명>` — 본 하네스대로 관찰 → 작성 → 린트 → 재생성.
- **기존 검토·재작성**: `/screen-def-review <SCR-ID>` — 기존 문서를 프로토타입과 대조해 격차 보고 후 골드 수준 재작성.
- **수기 지시 예시**(에이전트에게):
  > "`docs/screen-definitions/_작성하네스.md` 기준으로, 대상 화면의 프로토타입(HTML/JS)을 **직접 읽고** 관찰 목록을 먼저 만든 뒤 기획 관점만으로 화면 정의서를 작성해. §3 분기·엣지케이스와 §4 영역별 상태(빈/로딩/오류+단계상태)를 빠뜨리지 말고, 미확정은 §7에 화면-고유로 담당·기한과 함께. 코드 토큰은 빼고, 끝나면 `_lint.py`로 검증하고 `_build-data.py`로 재생성해."
