# 화면 정의서 — 의견청취 (대메뉴)

> 화면 ID: SCR-OPN-001 ~ 003 · 화면 유형: 목록·등록·상세(건의함) / 목록·상세·등록(위원회) / 목록·상세·등록(협의체) · 기획 관점(v6)
> 확정이 필요한 항목은 §7 미결사항에 모았습니다.
> 본 정의서는 의견청취가 **독립 대메뉴(GNB)로 승격**되고 처리 흐름이 **민원처리형(단일 흐름 CTA)** 으로 개편된 프로토타입 v2의 현재 구현 상태를 기준으로 하며, 하위 3개 화면(의견청취·건의함 / 산업안전보건위원회 / 협의체·점검표)을 한 문서로 정의한다.
> v1.2: 프로토타입 대조 검수 반영 — 점검결과지 미흡 문항의 담당부서·기한 지정과 종합 결과 자동 산출, 재처리 시 연동 초기화, 사진 첨부 제한, 반려 확인 절차 등을 현행화하고 화면-고유 미결을 §7에 추가. §6 중 "(개발 반영)" 표기는 확정 기획이나 현재 프로토타입에 아직 반영되지 않은 항목이다.

**목적·설명** — 종사자 의견 청취 절차(중대재해처벌법 시행령 제4조제7호)와 산업안전보건위원회 운영(산업안전보건법 제24조), 도급 시 안전·보건 협의체 운영·점검(산업안전보건법 제64조)을 하나의 대메뉴(3개 화면)에서 통합 관리하며, 조치가 필요한 건은 개선조치 메뉴로 자동 연계해 전사 PDCA 흐름으로 수렴시킨다.

- **의견청취·건의함**: 종사자 의견을 온라인·오프라인으로 접수하고, 접수대기 → 점검중/개선조치중 → 완료(+반려)의 민원처리형 흐름으로 처리·회신한다.
- **산업안전보건위원회**: 위원 구성, 의결 안건의 부서별 이행 현황, 회의록(작성·확정)을 관리한다.
- **협의체·점검표**: 협의체 위원 구성·운영 주기·운영 결과를 기록하고, 용역/구매설치/위탁사업/기타 구분의 점검결과지를 생성·결재한다.

메뉴 구조 변경 이력: 기존 "안전보건관리체계 > 의견청취(화면 내 3서브탭)" 구조에서 **GNB 대메뉴 + SNB 3화면**으로 승격되었다. 화면 내부 서브탭 바는 제거되었고 좌측 메뉴가 그 역할을 대신한다. 기존 주소(menu.html?m=opinion)로 진입하면 의견청취·건의함이 기본 표시된다(하위호환).

데이터는 세션 메모리에 보관되어 화면 이동 간 유지되며, 전체 새로고침 시 초기 예시 데이터로 리셋된다(프로토타입). 개선조치 연계 건은 개선조치 메뉴와 실시간 공유된다.

## 4. 화면 구성별 정리

### 4-A. 의견청취·건의함 (SCR-OPN-001)

#### 4-A-1. 상태별 카운트 바
- 구성: [전체 N] [접수대기 N] [점검중 N] [개선조치중 N] [완료 N] — 상태별 건수 버튼. **반려 건은 카운트 바에 별도 버튼이 없고**(전체 건수에는 포함) 목록의 상태 필터로 조회한다.

| 트리거 | 동작 결과 |
|---|---|
| 상태 버튼 클릭 | 목록이 해당 상태로 필터링(현재 선택 강조). [전체] 클릭 시 해제 |

- 상태: 건수는 현재 데이터 기준으로 실시간 집계된다.

#### 4-A-2. 의견 목록
- 구성: 검색 입력(제목·작성자, 입력 확정 또는 Enter 시 적용) + 분류/상태/부서 필터(부서에는 '미배정' 포함) + 목록 표 + 페이지네이션(총 건수·현재 범위 표시) + [+ 의견 등록].
- 컬럼: 분류(칩) · 제목(+안전점검/개선조치 연동 칩) · 작성자 · 담당부서(미배정 표시) · 등록일 · 상태(칩).

| 트리거 | 동작 결과 | 이동 / 메시지 |
|---|---|---|
| 행 클릭 | 상세 진입 | 4-A-4 |
| [+ 의견 등록] | 등록 폼 전환 | 4-A-3 |
| 검색·필터 변경 | 즉시 목록 갱신, 1페이지로 이동 | 0건 시 빈 상태 문구(E7) |

- 상태: 목록 → 상세 → 목록 복귀 시 필터·페이지 유지. 각 행의 업무 단계상태는 접수대기 → 점검중/개선조치중 → 완료(+반려) 칩으로 표시.

#### 4-A-3. 의견 등록 폼 (온·오프라인 병행)
- 구성: 작성자 정보(로그인 사용자·부서 자동 표시) / 의견 정보(분류 필수, 접수경로 온라인·오프라인, 제출자, 발생장소(글자수 카운터), 제목 필수(카운터)) / 상세 내용 필수(카운터) / 사진 첨부(드래그 영역, **JPG·PNG·GIF만, 최대 10MB·최대 5장** 안내).
- 제출자를 비우면 온라인은 로그인 사용자, 오프라인은 "(현장 종사자)"로 표기된다. 오프라인(서면·구두) 접수는 담당자가 대리 등록한다는 안내 문구 포함.

| 트리거 | 동작 결과 | 이동 / 메시지 |
|---|---|---|
| [등록하기] | 목록 최상단에 접수대기로 추가 + 담당자 접수 통지 알림(§6) + 이력에 "접수" 자동 기록 | "의견이 등록되었습니다" (오프라인이면 "담당자 대리 등록" 병기) |
| [취소] / [‹ 목록] | 저장 없이 목록 복귀 | |
| 사진 첨부 영역 | 프로토타입은 안내 표시로 대체 | |

- 상태: 신규 등록 건의 초기 상태는 접수대기, 담당부서는 미배정이다(조치 선택 시 담당자 소속으로 자동 배정).

#### 4-A-4. 의견 상세 — 2:1 배치
좌측: ① 의견 정보 카드(분류·상태 칩, 제목, 작성자/경로, 작성일, 담당부서, 발생장소, 상세내용, 첨부 사진) + ② 처리 카드. 우측: ③ 처리 이력 + ④ 연동 정보.

**② 처리 카드 — 상태별 5개 화면** (헤더에 "현재 상태" 칩 상시 표시)

| 상태 | 안내 배너 | 본문 구성 | 버튼 / 저장 동작 |
|---|---|---|---|
| 접수대기 | "접수 대기 중 — 의견을 검토하고 조치 방법을 선택하세요." | 조치 방법 3택 선택 카드: **현장점검 실시**(점검 종류 특별/일상/주간/합동/순회·점검 예정일(기본값 제공)·담당자·확인사항(의견 내용 미리 채움)) / **개선조치 바로 생성**(담당자·완료 예정일(기본값 제공)·개선 내용 필수(의견 제목 미리 채움)) / **반려**(사유 필수, "반려 사유가 기록되며, 되돌리려면 재처리해야 합니다" 경고). 선택 시 해당 폼만 펼침 | [저장] — 미선택 시 비활성. 저장 시 선택에 따라 점검중/개선조치중/반려 전이 + 담당부서 자동 배정 + 이력 자동 기록(반려는 확인창 통과 후) |
| 점검중 | "현장점검 진행 중 — 점검이 완료되면 자동으로 상태가 변경됩니다." | 연동 안전점검 카드(점검 종류·담당·예정일, [점검 상세 보기] — 안전점검 전용 메뉴 도입 전이라 안내 표시) | [경과 추가](이력 기록) · [완료 처리](처리 결과 입력 → 완료). 자동 전이는 미구현으로 수동 완료가 공식 신호(§6) |
| 개선조치중 | "개선조치 진행 중 — 개선조치가 완료되면 자동으로 상태가 변경됩니다." | 연동 개선조치 카드(제목·담당·예정일·조치중/종결 칩, [개선조치 상세 보기] → 개선조치 메뉴) | [경과 추가] · [완료 처리]. 연계 조치가 종결되면 **자동 완료 전이** |
| 완료 | "처리 완료 — 의견 처리 및 조치가 완료되었습니다." | 처리 완료일 · 처리자 · 처리 결과 요약 | [재처리] — 확인 후 접수대기로 복귀(연동·결과 초기화, E4) |
| 반려 | "반려 처리됨 — 반려 사유가 기록되었습니다." | 반려 사유 박스 | [재처리] |

**③ 처리 이력** — 세로 타임라인. 접수/현장점검 생성/개선조치 생성/경과/완료/반려/재처리가 단계 칩 + 내용 + 처리자 + 일시(연-월-일 시:분)로 자동 누적된다. 사용자가 직접 입력하는 항목은 "경과 추가"뿐이다.

**④ 연동 정보** — 연동 없음 시 "조치 선택 후 연동 정보가 표시됩니다" 빈 상태. 조치 후 안전점검/개선조치 미니 카드(배지·제목·담당·예정일·[바로가기]) 표시. 재처리 시 연동 카드가 사라지고 빈 상태로 복귀.

## 6. 정책

- **상태 용어 표준(5단계)**: 접수대기 → 점검중 | 개선조치중 → 완료 (+반려). 목록 필터·카운트 바·상태 칩·처리 카드가 동일 용어를 사용한다(디자인 토큰 문서 §5와 일치). 반려는 카운트 바에 별도 표기하지 않고 상태 필터로 조회한다.
- **민원처리형 단일 흐름**: 조치 방법 선택은 접수대기 상태에서만 가능하다. 진행 상태(점검중/개선조치중)에서는 [완료 처리]와 [경과 추가]만 제공하고, 완료/반려는 비가역이며 [재처리]로만 되돌린다. 재처리 시 연동 정보·반려 사유·처리 결과는 초기화되며, 이미 생성된 개선조치의 잔존 처리는 §7 OPN-10.
- **자동 전이**: 개선조치중 의견은 연계된 개선조치가 종결되면 완료로 자동 전이되고 이력에 자동 기록된다.
- **이력 자동 기록**: 접수·조치 선택·경과·완료·반려·재처리 등 모든 전이가 처리 이력에 자동 기록된다(일시·처리자 포함).
- **개선조치 연계**: "개선조치 바로 생성" 및 점검결과지 신규 등록의 미흡 문항은 개선조치 메뉴에 자동 등록되어 전사 PDCA 흐름으로 수렴한다. 점검결과지의 미흡 문항은 문항별로 지정한 담당부서·기한을 조치 항목에 반영한다.
- **점검결과지 종합 결과 자동 산출**: 미흡 문항이 1건이라도 있으면 「미흡」, 없으면 「적합」. 점검자가 임의로 종합 결과를 정할 수 없다.
- **결재 문서 보호**: 점검결과지는 결재중/승인완료 상태에서 수정·삭제가 잠긴다(편집 가능 상태는 미상신·반려). 승인완료 문서는 반려 후 수정하고, 반려 문서는 보완 후 재상신한다.
- **담당부서 자동 배정**: 접수 시 담당부서는 미배정이며, 조치 방법 저장 시 지정한 담당자의 소속 부서가 담당부서로 자동 배정된다.
- **오프라인 대리 등록 규칙(확정)**: 서면·구두 접수의 대리 등록은 의견청취 주관 부서(재난안전과 중대재해팀) 담당자만 수행한다. 제출자명은 실명 기재를 원칙으로 하되, 제출자가 원하지 않으면 "익명"으로 기재하고 원본 서면은 주관 부서가 보관한다. 대리 등록 건은 접수경로 "오프라인"으로 구분 표시된다.
- **현장점검 연동·완료 신호(확정)**: 안전점검 전용 메뉴 도입 전까지 현장점검 건은 의견 상세의 연동 카드로 관리하고, 점검 결과는 [완료 처리]의 처리 결과에 기록한다 — 즉 **담당자의 수동 [완료 처리]가 점검중 → 완료의 공식 완료 신호**다. 안전점검 전용 메뉴가 도입되면(기준문서함의 메뉴 승격 2차 검토와 연동) 점검 확정을 자동 전이 신호로 전환한다.
- **알림 시점·대상(확정)**: ① 접수 시 주관 부서 담당자에게 접수 통지 ② 완료 시 작성자에게 결과 회신 ③ 반려 시 작성자에게 사유 통지(개발 반영 — 현재는 반려 시 발송 없음) ④ 접수 후 7일 경과 미처리(접수대기 유지) 건은 담당자에게 재알림(개발 반영). 발송 채널은 §7 OPN-08에서 확정하며, 프로토타입은 문자 발송 안내로 시뮬레이션된다.
- **위원회 통계 집계 기준(확정, 개발 반영)**: "올해 회의 개최"는 당해 연도 회의록만 집계한다(현재는 전체 회의록 수 집계).
- **회의록 의결사항 연계 기본값(확정)**: 연계 등록 시 안건의 담당부서는 주관 부서, 이행기한은 회의일 기준 1개월로 자동 부여하고, 부서·기한 조정은 안건 부서이행 탭의 [수정]으로 한다.
- **데이터 보존(프로토타입)**: 의견·위원회·협의체 데이터는 세션 메모리 유지, 새로고침 시 초기화. 영구 저장은 개발 시 확정.
