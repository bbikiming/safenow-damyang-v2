# 화면 정의서 작성 규칙 (v6 · 기획 관점)

> 이 규칙은 `docs/screen-definitions/SCR-*.md` 에 자동 적용됩니다.
> 단일 출처: `_lint.py` · 자동 검사: `.claude/settings.json` PostToolUse 훅(`_lint_hook.py`).

## 1. 무엇을 쓰는가
"이 화면이 **무엇을 제공하는가**(기획 의도)"와 **화면 설계**를 업무 용어로 작성한다.
버튼이 무엇을 하고 어디로 가는지, 어떤 메시지·상태가 나오는지를 기술한다. **구현 방식은 개발이 정한다.**

## 2. 필수 구조 (7섹션 · 순서 고정)
1. 화면 정보 → 2. 권한 정보 → 3. 진입 / 이탈 경로 및 사용자 시나리오 → 4. 화면 구성별 정리 → 5. 유효성 검사 → 6. 정책 → 7. 미결사항

> **권한 연계**: §2 권한 정보는 권한을 직접 정의하지 않고 공통 권한 정의서(`_공통_권한정의.md`) §6 화면별 권한 매핑의 `[SCR-ID]` 행을 참조한다. 권한 확정·변경은 그 문서에서만 하고(단일 출처), 새 화면 정의서를 작성하면 §6에 그 화면의 행을 추가한다.

## 3. 금지 — 코드 수준 내용
기획 문서에는 다음을 쓰지 않는다(자동 검사가 차단):
- 함수·메서드명(`PG.x`, `EDOC.x`, `renderBody()` 등), 내부 전역/데이터 구조명(`DY_DOCS`, `CHECKLIST_PRESETS`, `APPR` 등)
- DOM 요소 ID(`#policy-target`), CSS 클래스(`sl-...`, `chip-mini`), `data-*` 속성
- 스토리지 API/키(`localStorage`, `dy-edoc-v1`)
- 코드 연산자·수식·메서드 체이닝(`===`, `||`, `.indexOf(`, `% 9000`, `× 137`)
- 소스 파일:라인 참조(`menu.js L12`), 보안/코드품질 분석(`XSS`, `데드코드`, `stale`, `try-catch`)
- 데이터셋 실측 건수 나열은 지양 — 수치보다 기획 의도/규칙을 적는다.

> URL 경로(`docs-preset.html`, `menu.html?m=policy`), PDCA·세트·처리유형 같은 **업무 개념**은 허용된다.

## 4. `[확인필요]`는 §7 미결사항에 집결
- 확정이 필요한 항목은 본문에 흩지 말고 **§7 미결사항 표**에 모은다.
- §7 표 컬럼: `ID | 항목 | 확인 내용 | 담당 | 기한` — **기한 필수**.

## 5. 검사 방법
- 로컬: `python3 docs/screen-definitions/_lint.py [파일...]`
- 자동(작성 즉시): Write/Edit 시 훅이 검사 → 위반하면 수정 피드백.
- 빌드: `python3 docs/screen-definitions/_build-data.py` 가 재생성 시 위반 요약을 함께 출력.

## 6. 새 규칙 추가/완화
규칙은 `_lint.py` 의 `REQUIRED_SECTIONS` / `FORBIDDEN` 한 곳에서 관리한다. 수정하면 훅·빌드·로컬 검사에 모두 반영된다.
