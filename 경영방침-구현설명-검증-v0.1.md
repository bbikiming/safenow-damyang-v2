## 1. 왜 이렇게 만들었나 (핵심 논리)

경영방침은 단순한 "문서 한 장"이 아니라 **법이 요구하는 하나의 순환(PDCA)**입니다.

> **세우고(Plan) → 결재받고 → 알리고(Do) → 점검하고(Check) → 고친다(Act)**

이 순환이 끊기면 법적 의무 미이행이 됩니다. 그래서 화면을 이 순환의 **순서 그대로 4개 카드**로 배치했습니다. 사용자는 위에서 아래로 따라가기만 하면 한 사이클이 완성됩니다.

```
 ① 목표·방침 세우기 ──▶ ② 근거·고려사항 ──▶ ③ 점검표 준비 ──▶ ④ 반기 점검·개선
   (Plan)                 (왜 해야 하나)        (무엇을 볼까)       (Check→Act)
```

또 하나의 원칙은 **v2 전체 철학(지자체 사용성)** 입니다.
- 대메뉴 클릭 → 바로 "할 일"이 보인다 (진입 1뎁스)
- 탭은 [업무 기능] / [문서] 2개만
- 복잡한 편집(법령·점검표 만들기)은 **모달**로 빼서 평소 화면은 단순하게 유지

## 2. 법(SFR-005)이 요구한 것 → 화면 기능 (왜 각 기능이 있나)

| 법이 요구하는 것 | 근거 | 화면에서 어디에 | 왜 필요한가 |
|---|---|---|---|
| 안전보건 목표·경영방침 설정 | 중처법 시행령 §4-1 | ① 카드 (방침·목표 입력) | 경영책임자(군수)의 의지 선언 = 모든 안전활동의 출발점 |
| 설정 가이드 제공 | 산안법 §14 | ① 카드 `설정 가이드` 버튼 | 담당자가 "무엇을 어떻게 써야 하나" 막히지 않도록 |
| 관련 근거·법령 관리(CRUD) | 중처법 시행령 §4 | ② 카드 `근거·고려사항 관리` | 방침·점검 항목이 "어느 법 때문인지" 추적 가능해야 함 |
| 설정·점검 시 고려사항 관리 | 중처법 시행령 §4-1 | ② 카드 동일 모달 | 매년 같은 실수 반복 방지 (전년 결과 반영 등) |
| 근거 ↔ 세부내용 매칭 | — | ②·③ 카드 (항목별 근거) | 점검 결과를 법 조항에 바로 연결 |
| 점검표 등록 (O/X·텍스트, 근거 직접등록) | — | ③ 카드 `점검표 편집` | 무엇을 점검할지 사전 정의 = 점검의 일관성 |
| 점검자 지정 → 전자결재 승인 | 중처법 시행령 §4-1 | ① 점검자 + ④ 온나라 상신 | 점검의 책임 소재 + 공식 승인 |
| 증빙자료 다중 첨부 | — | ① 증빙 + ④ 점검표 첨부 | 점검을 "했다"는 객관적 증거 보존 |

> 결론: 화면의 4개 카드·각 버튼은 **장식이 아니라 SFR-005의 각 줄을 1:1로 구현한 것**입니다.

## 3. 화면 구성 — 4개 카드가 하는 일

### ① 안전·보건 목표와 경영방침 (세우기)
- **입력**: 경영방침 문구 · 안전보건 목표 · 점검자 지정 · 증빙자료(다중 첨부)
- **버튼**: `설정 가이드`(작성 4원칙 안내) · `저장` · `온나라 결재 상신` · `공표(전 직원 알림)`

### ② 근거법령·고려사항 (왜 하는지)
- **표시**: 법령↔세부내용 매칭 5건 + 고려사항 요약
- **버튼**: `근거·고려사항 관리` → 모달에서 등록·수정·삭제

### ③ 경영방침 점검표 관리 (무엇을 볼지)
- **표시**: 영역 · 점검항목 · **결과유형(O/X·텍스트)** · 관련 근거
- **버튼**: `점검표 편집` → 모달에서 항목·결과유형·근거 편집

### ④ 수립·이행 점검 (반기) (점검·개선)
- **버튼**: `이행점검 작성` → 점검표 작성(O/X·텍스트) → **X 항목 자동 개선조치** → 온나라 결재 상신

## 4. 데이터가 어떻게 들어가고 흐르는가

### 4-1. 어떤 데이터가 들어가나

| 구역 | 들어가는 데이터 | 형태 |
|---|---|---|
| ① 설정 | 경영방침(문장), 목표(여러 줄), 점검자(이름·부서), 증빙(파일 여러 개) | 텍스트 + 파일 |
| ② 근거·고려사항 | 법령명·조항·세부내용 / 고려사항 구분(설정·점검·공통)·내용 | 목록(표) |
| ③ 점검표 | 영역·항목·결과유형(O/X·텍스트)·관련근거 | 목록(표) |
| ④ 반기 점검 | 점검일·점검자·항목별 결과(O/X/해당없음)·비고·증빙 | 점검표 1건 |

### 4-2. 어디에 저장되나

- **지금(프로토타입)**: 브라우저 임시 저장소(localStorage)에 보관 → 데모 중 화면을 옮겨도 유지됩니다.
  - `dy-edoc-v1` = 작성한 전자문서(방침·점검표) · `dy-imp-v1` = 자동 생성된 개선조치 · `dy-ntf-v1` = 발송 알림
- **실제 구축 시**: 서버 DB에 저장 (필드·키는 개발자 영역 — 별도 [데이터 확인 문서](경영방침-SFR-005-데이터확인-v0.1.md) 참조)

### 4-3. 전체 흐름 (한 사이클)

```
 [작성]                  [결재]                [공표]              [점검·환류]
 방침·목표 입력      →   온나라 결재 상신   →   전 직원 알림   →   반기 점검표 작성
 점검자 지정             (문서 → 확정)          (게시판+문자)        │ 항목별 O/X·텍스트
 증빙 첨부                                                          │
                                                                   ├─ O(적합) ──▶ 완료
                                                                   └─ X(부적합) ─▶ 개선조치 자동 등록
                                                                                  └▶ (개선조치 메뉴로 유입)
                                                                                     접수→계획→진행→완료확인→종결
```

### 4-4. 문서 상태의 변화 (전자문서 공통 규칙)

```
 작성중 ──[등록]──▶ 등록완료 ──[확정]──▶ 확정(=온나라 상신)
   (수정 자유)       (확정 전 수정 가능)     (잠금 · 개정으로만 변경)
```
- **확정 시 두 가지가 자동 발생**: ㉠ 온나라 결재 상신 ㉡ 점검표의 X 항목 → 개선조치 자동 등록

## 5. 온나라 연계 — 어디서, 어떻게

### 5-1. 연계 지점은 딱 2곳

| # | 위치 | 무엇을 상신 | 트리거 |
|---|---|---|---|
| 1 | ① 카드 `온나라 결재 상신` 버튼 | **경영방침·목표** 결재 | 버튼 클릭 |
| 2 | ④ 점검표 `확정 · 온나라 결재 상신` | **반기 점검 결과** 결재 | 점검표 확정 |

### 5-2. 작동 방식

```
 담당자 ─[상신 클릭]─▶ 시스템                                   온나라 전자결재
                       │ 문서번호 발번 (예: 온나라-2026-3174)   ┌─────────────┐
                       │ 결재선: 팀장 → 과장 → 부군수      ───▶ │ 팀장 → 과장   │
                       │ 상태: 등록완료 → 확정                  │  → 부군수     │
                       ▼                                       └──────┬──────┘
                  "상신 완료" 안내 팝업                                │ 결재 완료
                                                                      ▼
                                                  결재 결과 콜백 ─▶ 시스템 문서 상태 자동 갱신
```

### 5-3. 지금 vs 실제 구축

| 구분 | 지금 (프로토타입) | 실제 구축 시 (확정 필요 🔴) |
|---|---|---|
| 상신 | 안내 팝업 1회 (시뮬레이션) | 온나라 API로 실제 상신 |
| 문서번호 | 화면에서 임시 생성 | 온나라가 부여한 번호 수신·매핑 |
| 결재 완료 | 표현만 | 결재 완료 콜백 → 시스템 상태 자동 갱신 |
| 결재선 | 팀장→과장→부군수 (고정) | 실제 직제/결재 규정과 일치 확인 |

> **핵심 결정 사항(발주처)**: 온나라를 ① API 직접 연계 / ② 결재 사본 첨부 / ③ 수동 상신 중 어떤 방식으로 할지. 이게 정해져야 ④의 "확정→상신"을 실제로 구현할 수 있습니다.

## 6. UI/UX 검증 — 이게 올바른가?

### 6-1. 평가 기준
① SFR-005 요구 충족 ② v2 설계 철학(지자체 사용성) ③ 법적 업무 흐름 일치 ④ 일반 UX 원칙 ⑤ 접근성

### 6-2. 항목별 판정

| 기준 | 판정 | 근거 |
|---|---|---|
| SFR-005 전 항목 커버 | ✅ 양호 | 설정·가이드·근거·고려사항·점검표·점검·증빙·결재·개선 모두 구현 |
| 진입 깊이 최소화 | ✅ 양호 | 대메뉴 → 1클릭 진입, 할 일이 즉시 노출 |
| 탭 단순화 | ✅ 양호 | [업무 기능]/[문서] 2탭 원칙 준수 |
| 법적 흐름 일치 | ✅ 양호 | 카드 순서 = 작성→결재→공표→점검→개선 (PDCA) |
| 자동 연계 | ✅ 우수 | 점검 X → 개선조치 자동 등록 (수기 누락 방지) |
| 근거 추적성 | ✅ 양호 | 모든 점검항목에 법적 근거 표기 |
| 화면 일관성 | ✅ 양호 | 다른 대메뉴와 동일한 카드·모달·전자문서 엔진 재사용 |
| 접근성(기본) | 🟡 보통 | 모달에 `role=dialog`·`aria-modal`·ESC·배경클릭 닫기 있음 / **포커스 트랩 없음** |

### 6-3. 잘된 점 (이대로 유지)
1. **순서가 곧 가이드** — 위에서 아래로 따라가면 법적 사이클 1회 완성.
2. **점검 → 개선 자동 연결** — X 항목이 사람 손 거치지 않고 개선조치로 넘어감 = 누락·증빙 공백 방지.
3. **편집은 모달로 격리** — 평소 화면은 보기 좋게, 만들 때만 깊게.
4. **근거 상시 노출** — 감사·점검 시 "어느 법 때문인지" 바로 제시.

### 6-4. 보완점 (실 구현 전 처리 권장 · 우선순위)

| 우선 | 항목 | 내용 | 영향 |
|---|---|---|---|
| 1 | **점검표 단일 출처화** | ③ 모달의 항목과 ④ 실제 점검표 항목이 현재 별도 데이터. 하나로 묶어야 "편집→점검"이 실제로 연결 | 기능 신뢰성 |
| 2 | **텍스트형 결과 런타임** | ③에서 "텍스트"로 지정해도 ④ 점검 시 O/X 버튼으로만 표시됨. 텍스트 항목은 입력칸으로 분기 필요 | SFR 정합 |
| 3 | **KPI 항목화** | 목표가 자유 텍스트라 자동 집계 불가. 측정지표로 항목화하면 현황 KPI와 연결 | 통계 연계 |
| 4 | **온나라 실연계 방식 확정** | §5-3 — API/사본/수동 결정 | 결재 동작 |
| 5 | **모달 포커스 트랩** | 키보드 사용자가 모달 밖으로 탭 이동되지 않게 | 접근성 |

### 6-5. 종합 판정

> **방향: 올바름 ✅.** 화면은 SFR-005의 의도(법적 PDCA 사이클)를 정확히 표현하고, v2의 지자체 사용성 철학에 부합합니다. 프로토타입으로서 "무엇을·왜·어떤 순서로" 만들지를 명확히 보여줍니다.
>
> 단, **실제 구축 착수 전 6-4의 1~4번(점검표 단일화·텍스트형·KPI 항목화·온나라 연계)**을 확정해야 화면의 약속이 실제 동작으로 이어집니다. 이 4건은 모두 "데이터·정책 결정"이지 화면 재설계가 아니므로, **현재 UI/UX 구조는 그대로 두고 진행**해도 됩니다.
