## 1. REQ-A — 점검 대상 관리 (기관·시설)

### 데이터 (js/bgt.js)
```
targets: {
  org: [{ id:'tg-o<n>', name:'재난안전과', src:'수기' }],            // 기관 = 시스템 설정 등록값
  fac: [{ id:'tg-f<n>', name:'담양군민체육관', src:'FMS'|'수기', syncedAt }], // 시설 = FMS 연계 + 수기 보완
}
```
- 시드: 기관 10건(기존 부서 목록, src 수기), 시설 6건(기존 6종, src FMS·syncedAt '2026-07-01 06:00').
- FMS 마스터(연계 시뮬레이션용 상수, bgt.js 내부): 기존 6건 + 신규 3건(담양청소년수련관, 담양국민체육센터, 메타세쿼이아랜드 관리동) = 9건.

### API (DYBGT 신설)
- `listTargets(type)` → type '기관'|'시설' → 해당 배열
- `addTarget(type, name)` → `{ok, msg, id}` — 공백·중복명 차단
- `removeTarget(type, id)` → `{ok, msg}` — **점검표(sheets)에서 target으로 사용 중이면 차단**
- `syncFMS()` → `{ok, added, total, at}` — FMS 마스터와 병합(이름 기준, 신규만 추가·기존 유지), 전 시설 syncedAt 갱신

### 화면
- **BGT02-S 섹션 C '점검 대상 관리'** (섹션 B 아래): 기관 블록 + 시설 블록.
  - 각 블록: 목록(이름 + src 칩 + 점검표 사용 중 뱃지 + [삭제]) + [+ 등록] 인라인 입력(섹션 B와 동일 UX, 모달 금지)
  - 시설 블록 헤더: [FMS 연계 동기화] 버튼 + '마지막 동기화 {syncedAt}' 표기. 클릭 → syncFMS() → 토스트 "FMS 시설 {total}건 동기화 — 신규 {added}건" + 재렌더
  - src 칩: FMS=`chip-mini wt-elec`, 수기=`chip-mini wt`
- **BGT01-V 생성 모달**: `DEPTS`·`FACILITIES` 하드코딩 상수 **삭제** → 대상 select를 `listTargets(구분)`으로 구성. 대상 0건이면 select 대신 안내문 "예산 기준 설정 > 점검 대상 관리에서 대상을 먼저 등록하세요."
- 연계 관리(admin-integration.html)의 기존 FMS 카드(일 1회 동기화)와 서사 일치 — 수동 [FMS 연계 동기화]는 즉시 반영용. admin-integration.html은 **수정하지 않는다**.

## 2. REQ-B — 이모지 전면 제거

### 규칙
- **제거**: 그림 이모지 전부 — U+1F300~1FAFF (예: 검수 스캔에서 확인된 📋 💰 📈 📎 📨 📄 🔍 🔎 🔒 🔗 🖼 🏛 👤 📅 📍 🧾 🖥 📂 💾 📦 🚨 🎉 🎓 💡 🛠 🛡 🚜 🚪 🧪 🔖 📜 🌡 💨 🔥) + 이모지 표현 기호 ⚠ ⛔ ⚖ ⚡ ❄ ⬇ ℹ ⚪ 🔵 🔶 🟢 ✏ + 잔여 variation selector(U+FE0F).
- **유지**: 타이포그래피 기호 — 화살표(→ ↔ ← ↑ ↓ ↗ ↳ ↻ ⇒), 체크·엑스(✓ ✕ ×), 별점(★ ☆).
- **대체 3원칙**: ① 인접 텍스트가 이미 의미를 전달하면 이모지만 삭제 ② 이모지가 유일한 의미 전달자면 짧은 텍스트 라벨로 대체(예: 📎 파일명 → 파일명, 🔍 확대 → 확대) ③ 색 인디케이터(🔵🔶🟢⚪ 위험등급 등)는 CSS 원형 도트(`.link-status-dot` 유사 span)로 대체 — 색 의미 보존.
- KPI 카드 아이콘 슬롯(`kpi-card-icon`)은 이모지 대신 **슬롯 자체 제거** (kpi-card는 현재 bgt.js만 사용).

### 스코프
- **포함**: 전 런타임 화면 — 루트 *.html(아래 제외 항목 빼고 전부) + js/*.js (bgt·common·edoc·edoc-templates·menu·layout·edu·doc-detail·doc-register·setlist·risk-assessment·risk-checklist·risk-checklist-data·risk-process-eval·risk-tour·screen-definitions.js 등)
- **제외**: screen-defs-data.js(빌드 생성물 — 원본 `_build-data.py` 경로에서 별도 처리), sets-data.js·sets-data-v2.js·risk-sif-data.js(★뿐이라 대상 없음), onepager-docflow.html(기획 문서 페이지)

## 3. REQ-C — 온나라 결재 수신 구조 + 결재 이력

### 상태 표준화 (기존 시스템 정합)
- sheet.status: `작성중` / `결재중`(구 '상신') / `승인완료`(구 '승인') / `반려`
- 결재 칩: `chip-status` + 색 매핑 작성중=neutral · 결재중=info · 승인완료=success · 반려=danger (경영방침 `apprColor` 표준). 목록 테이블·상세 헤더 공통.
- 편집 잠금 매트릭스(1차 검수 §3.1)는 명칭만 치환하고 규칙 유지: 편성액=작성중·반려 / 집행액·비고=결재중 제외 항상 / 삭제=작성중만.

### 데이터
```
sheet.onnara = { docNo:'온나라-2026-<4자리>', steps:[{ at, act, by, note }] }
```
- `submitSheet(id)`: status '결재중' + docNo 발급·저장(최초 상신 시 1회) + steps에 `['결재 상신', '김담당(재난안전과)']` 행 추가. 재상신도 행 누적.
- **approveSheet/rejectSheet 제거** → `receiveOnnara(id, kind, reason)` 신설 (결재 결과는 온나라에서 수신하는 값):
  - kind `'mid'`: 중간 결재 1단계 승인 수신 — steps의 승인 진행도에 따라 '팀장 승인'(이팀장) → '과장 승인'(김과장) 순차 1행 추가, status는 '결재중' 유지
  - kind `'approve'`: 최종 승인 수신 — 미진행 중간 단계 행 자동 보충 + '부군수 승인 — 승인 완료'(부군수) 행 추가, status '승인완료'
  - kind `'reject'`: 반려 수신 — '반려 — {reason}'(당시 결재 단계 처리자) 행 추가, status '반려' + rejectReason 저장. reason 필수.
- 결재선 고정: 팀장 → 과장 → 부군수 (온나라 팝업 문구와 동일).
- sheet.history(처리 이력)에도 상신/수신 이벤트 기록 유지.

### 화면 (BGT01-V 상세 '전자결재' 카드)
- 공통 행: `온나라 결재` 라벨 + 결재 칩 + 문서번호(상신 후) + **[이력]** 버튼 — 경영방침 상세의 결재 행과 동일한 배치.
- 상태별:
  - 작성중·반려: [온나라 결재 상신] → `EDOC.onnaraPopup`(발급 docNo 표기) → 확인 시 결재중 전이
  - 결재중: [결재 상태 조회] 버튼 추가 → 모달 **'온나라 결재 상태 조회 (연계 시뮬레이션)'**: 현재 steps 4열 테이블 + 하단 모의 수신 버튼 3개 `[중간 승인 수신]` `[최종 승인 수신]` `[반려 수신]`(반려 사유 입력 필드 — 모달 내 인라인, 필수). 버튼 클릭 = 연계 응답이 수신된 것으로 반영 + 토스트. **시스템 화면 어디에도 승인/반려 '처리' 권한 버튼은 없다** — 결과는 항상 온나라 수신값.
  - 승인완료: 칩 + [이력] + 안내문(집행액·비고 상시 입력 가능)
- **[이력] 모달**: 제목 `온나라 결재 이력 — {docNo}` (미상신이면 docNo 생략), `table-figma` 4열 **일시 / 처리 / 처리자 / 비고**, rows = `sheet.onnara.steps` 실데이터 동적 생성. steps 없으면 "결재 이력이 없습니다 — 상신 후 이력이 기록됩니다." + footer [확인] — menu.js `opnChkApprHistory`(의견청취)와 동일 포맷.
- 기존 상세 하단 '결재 이력' 카드는 **'처리 이력'으로 개칭** (문서 생명주기 타임라인 — 온나라 결재 이력 모달과 역할 구분).

### 시드·마이그레이션
- localStorage 키 `'dybgt-v1'` → **`'dybgt-v2'`** (구 키는 무시 — 상태값 개편에 따른 프로토타입 마이그레이션 방식).
- 시드 반영: 승인완료 2건 = docNo + steps 4행(상신/팀장/과장/부군수), 결재중 1건 = docNo + steps 1행(상신), 작성중 1건 = onnara null. targets 시드 포함(§1).

## 4. 수용 기준 (AC — 통합 검수 체크리스트)

1. 생성 모달 대상 목록이 `listTargets()`에서 옴 — bgt.js에 DEPTS/FACILITIES 상수 부재
2. [FMS 연계 동기화] → 신규 시설 3건 추가·토스트·syncedAt 갱신, 생성 모달에 즉시 반영
3. 점검표에서 사용 중인 대상 삭제 차단(토스트), 대상 등록 중복명 차단
4. §2 스캔 스크립트 스코프 내 그림 이모지 **0건**, 화살표·별점·체크 기호는 잔존 허용
5. 상신 → 결재중(문서번호 저장·표시), [결재 상태 조회] 모달에서 중간/최종/반려 수신 3종 모두 반영 가능, 시스템 내 승인·반려 처리 버튼 부재
6. [이력] 모달이 steps 실데이터로 동적 렌더(4열 표준), 미상신 시 안내문, 반려→재상신 시 행 누적
7. 결재 칩이 `chip-status` 표준 4색, 편집 잠금 매트릭스 규칙 유지(명칭만 치환)
8. 키 dybgt-v2 신규 시드 정상 부팅, 콘솔 오류 0, 1차 검수 통과 흐름(자동 저장·합계 재계산·단일 모달) 회귀 없음
9. **기획 순수성** — 프로토타입은 구현 영역을 침범하지 않는다:
   - ① 화면에 노출되는 모든 문구에 구현 용어(스토리지·API·함수·클래스·파일명 등) 부재 — "(프로토타입)"·"(연계 시뮬레이션)" 같은 경계 표기는 허용
   - ② 모든 동작이 기획 요소·유효성 규칙·화면 플로우만으로 완결 기술 가능(버튼→결과→차단 메시지가 화면 안에서 설명됨)
   - ③ 화면 정의서에는 구현 참조를 싣지 않는다 — 검수 리포트의 구현 추적 정보(클래스·API 매핑)는 내부 검수용으로 분리 유지

# 3차 변경 — 점검 대상을 '연계 선택(팝업 뷰)'으로 전환 (REQ-A 개정)

- 배경: '중대재해 관리 대상 현황정보 관리' 요구사항 확인 — **대상 관리 메뉴가 별도 구축 예정**(FMS 시설물을 엑셀 데이터로 불러와 업데이트, 안전점검 등급·시설물안전법 1·2·3종 분류 자동 연계, FMS 변동 시 자동 현행화). 따라서 예산의 점검 대상 관리는 자체 등록·자체 FMS 동기화를 없애고 **연계 원본에서 선택**하는 구조로 바꾼다.
- 소스: **시설 = 대상 관리(관리대상 현황) 데이터 연계** / **기관 = 타 메뉴 공용 조직도 기반** (※ 기관 범위는 고객사 확인 필요 — 미확정 플래그, 프로토타입은 조직도 부서 기준으로 진행)
- 추가 방식: 둘 다 **팝업 뷰에서 선택 후 추가** (다중 선택).

## 7. 설계

### 데이터·API (js/bgt.js)
- 제거: `FMS_MASTER`, `syncFMS`, 공개 `addTarget`(수기 등록 폐지)
- 신설 소스:
  - 시설 원본 `FAC_SOURCE`(대상 관리 연계 데이터 시뮬레이션, 9건): `{ name, cls:'1종'|'2종'|'3종', grade:'A'~'D' }` — 요구사항의 FMS 속성(종별·안전점검 등급) 포함
  - 기관 원본: `window.EDOC.ORG_TREE`의 부서 목록(타 메뉴 공용 조직도 — 6개 부서) 참조. EDOC 부재 시 빈 목록 폴백
- 신설 API:
  - `listSources(type)` → 기관: `[{name, meta:'구성원 N명'}]` / 시설: `[{name, cls, grade}]`
  - `addTargetsFromSource(type, names[])` → `{ok, added, skipped}` — 이미 등록·원본에 없는 이름은 건너뜀
- `targets` 항목: 기관 `{id, name, src:'조직도'}` / 시설 `{id, name, src:'대상관리', cls, grade}` (종별·등급을 복사해 화면 표기)
- 시드 축소(픽커 데모 목적): 기관 2건(재난안전과·환경과) + 시설 2건(담양군민체육관·담양하수처리장) — 점검표 시드가 참조하는 대상만 선택된 상태로 시작. 스토리지 키 `dybgt-v3`.
- `listTargets`·`removeTarget`(사용 중 가드)·`targetUsed`·생성 모달 연동은 유지.

### 화면 (BGT02-S 섹션 C 개편)
- 블록 헤더: 기관 = "조직도 연계" 라벨 + [+ 기관 추가] / 시설 = "대상 관리(FMS) 연계" 라벨 + [+ 시설 추가]. 인라인 수기 입력·[FMS 연계 동기화]·'마지막 동기화' 표기 제거.
- **선택 팝업 (단일 모달 — DYV2.openModal, 화면당 이 모달 1종)**:
  - 기관: 제목 '기관 선택 — 조직도', 검색 입력 + 부서 체크 목록(부서명 + 구성원 수). 
  - 시설: 제목 '시설물 선택 — 대상 관리 연계', 검색 입력 + 체크 목록(시설명 + 종별 칩 + 안전점검 등급 칩).
  - 공통: 이미 등록된 항목은 체크 표시 + 비활성 + '등록됨' 표기. 하단 [취소] [선택 추가 (N건)] — N은 체크 수 실시간 갱신, 0건이면 토스트 차단. 추가 성공 토스트 "{N}건이 추가되었습니다."
- 시설 행: 이름 + 종별 칩 + 등급 칩 + (점검표 사용 중 뱃지) + [삭제] / 기관 행: 이름 + (사용 중 뱃지) + [삭제]
- 안내 노트 갱신: "점검 대상은 조직도(기관)와 대상 관리(시설물)에서 선택해 추가합니다. 시설물 현황은 대상 관리 메뉴에서 FMS 연계로 자동 현행화되며, 점검표에서 사용 중인 대상은 삭제할 수 없습니다."

## 8. 수용 기준 (AC-3차)

1. 섹션 C에 수기 입력·FMS 동기화 버튼 부재 — 추가 경로는 팝업 선택뿐
2. 기관 팝업 원본 = 공용 조직도 부서 목록 / 시설 팝업 원본 = 대상 관리 연계 목록(종별·안전점검 등급 표기)
3. 이미 등록 항목 비활성 처리, 다중 선택 일괄 추가, 선택 0건 차단, 중복 등록 불가
4. 선택된 시설 행에 종별·등급이 표기됨 (FMS 속성 전달)
5. 생성 모달은 '점검 대상 관리에 추가된 대상만' 노출 (연동 유지), 사용 중 대상 삭제 가드 유지
6. 단일 모달 규칙·콘솔 오류 0·기획 순수성(AC9) 유지
7. 스토리지 `dybgt-v3` 신규 시드 (기관 2·시설 2 선택 상태로 시작)
8. (문서) 기관 대상 범위는 고객사 확인 필요 — 미확정 사항으로 검수 리포트·정의서에 플래그
