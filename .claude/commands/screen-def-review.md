너는 기존 **화면 정의서**를 검토하고 골드 스탠다드(SCR-DOC-007) 품질로 재작성한다. 대상: **$ARGUMENTS**

1. **하네스·기준 읽기** — `docs/screen-definitions/_작성하네스.md`, `_규칙.md`, `_화면정의서_템플릿_v6.md`, 골드 `SCR-DOC-007_업무등록마법사.md` 를 읽는다.

2. **현행 문서 + 프로토타입 대조** — 대상 SCR의 기존 정의서와, 그 화면의 실제 프로토타입(HTML/JS)을 **직접 읽어** 다음 격차를 찾는다:
   - 프로토타입에는 있는데 문서에 **없는** 동작·영역·진입점(누락)
   - 문서에는 있는데 프로토타입과 **어긋나는** 서술(예: 화면이 갱신되어 더는 맞지 않는 설명)
   - 골드 수준 대비 **얕은** 섹션(§3 시나리오·§4 상태·§5·§6·§7 깊이 부족)
   먼저 격차 목록을 간단히 보고한다.

3. **재작성** — 위 격차를 모두 반영해 7섹션을 골드 수준으로 다시 쓴다. 기획 관점만, 코드 토큰 금지, 미확정은 §7(담당·기한). 관련 화면 SCR-ID 상호참조.

4. **검증·재생성** — `python3 docs/screen-definitions/_lint.py <파일>` 위반 0 확인 → `python3 docs/screen-definitions/_build-data.py` 재생성. 권한 매핑 행이 없으면 `_공통_권한정의.md` §6에 추가.

5. **자가검증·보고** — `_작성하네스.md` §4 체크리스트를 점검하고, 이번에 바로잡은 격차를 요약 보고한다.

원칙: 프로토타입이 진실의 출처다. 문서가 프로토타입과 어긋나면 문서를 고친다. 구현 방식은 쓰지 않는다.
